## Question 1 — DeepSeek V3.1 Q4_K_M 18.30 t/s: exact n-cpu-moe value and batch scaling

### Verified numbers from discussion #18005

The DeepSeek V3.1-Terminus Q4_K_M number was generated by `llama-bench`, not
`llama-server`. The exact reproducible command from the thread:

```
GGML_CUDA_ENABLE_UNIFIED_MEMORY=1 \
./bin/llama-bench \
  -m ~/fairydreaming/models/DeepSeek-V3.1-Terminus-Q4_K_M-00001-of-00009.gguf \
  -fa 1 \
  -d 0,4096,8192,16384,32768,65536 \
  -p 2048 \
  -n 32 \
  -ub 2048
```

**There is no `--n-cpu-moe` value in the command line.** The expert-on-CPU
placement is done by the **cudaMemAdvise patch** applied at build time, which
targets all tensors matching `blk.*.ffn_(up|gate|down)_exps.weight`. Every MoE
layer's experts go to CPU. With DeepSeek V3.1 architecture that's effectively
the equivalent of `--n-cpu-moe 58` (3 dense layers + 58 MoE layers; experts in
the 58 MoE layers all land on CPU). Round 1's claim of "--n-cpu-moe 58" was
inferred, not directly stated in the bench command — the patched binary
achieves the same placement without the flag.

### Updated benchmark table (depth-stratified, verified verbatim from #18005)

DeepSeek V3.1 Q4_K_M (671 B), GH200 144 GB HBM3e, CUDA 13.0, rev
`c6f6e4f96a7f7bce49f5c21d19ee69fb8b72f84d`:

| Mode        | Metric          | Value                  |
|-------------|-----------------|------------------------|
| UM, no patch| pp512           | 41.63 ± 0.38           |
| UM, no patch| tg128           | 8.86 ± 0.14            |
| UM + patch  | pp512           | 281.78 ± 3.20          |
| UM + patch  | tg128           | **18.36 ± 0.01**       |
| UM + patch  | pp2048 @ d65536 | 110.51 ± 0.23          |
| UM + patch  | tg32 @ d65536   | 15.69 ± 0.05           |

Note round 1 wrote 18.30; the precise figure is **18.36 ± 0.01 t/s**. The
patch yielded a **2.07x TG speedup and ~6.77x PP speedup** vs naive UM.

### Batch scaling — what we know vs what we don't

The bench command above tests a single fixed `-ub 2048 -p 2048 -n 32`. The
thread does **not** include a sweep over batch sizes for DeepSeek V3.1.
However, the GH200 thread does include `llama-batched-bench` runs for
GPT-OSS 120B with `-npp 512,4096,8192 -ntg 32 -npl 1,2,4,8,16,32`. The
qualitative finding from this and adjacent guides:

- **`--n-cpu-moe` interacts strongly with `-b`/`-ub`.** Defaults (`-b 2048
  -ub 512`) are too small for MoE-offload workloads; the prompt-processing
  path benefits from copying CPU-resident expert weights to VRAM in larger
  micro-batches.
- Recommended `-b 4096 -ub 4096` for MoE+offload (Doctor-Shotgun guide). On
  GH200 the thread used `-ub 2048` and got 110 t/s PP at d=65536 — likely
  more headroom at 4096.
- The offload threshold is governed by `GGML_OP_OFFLOAD_MIN_BATCH` (default
  32 tokens). For very-large CPU weight allocations the guide recommends
  hundreds of tokens; on GH200 the C2C bandwidth (135 GiB/s measured async
  memcpy) makes the offload trip cheaper than on PCIe.
- TG (single-stream decode) is **not** very batch-sensitive at low concurrency
  because only the active experts move; the gain from larger `-ub` shows up
  in PP and in multi-stream `-npl > 1` runs.

**Gap remaining**: no public batch sweep with `--n-cpu-moe` on GH200 for
DeepSeek V3.1. The data exists only for GPT-OSS 120B (all-GPU) on GH200.

### What's in the `-d` column

The `-d 0,4096,...,65536` flag is **depth/context-position**, not batch.
It pre-fills that many tokens of dummy KV before measuring, so the
"@ d65536" rows are TG decode at 64K context — that's how
15.69 t/s @ d=65536 vs 18.36 t/s @ d=0 shows attention scaling cost.

## Question 2 — cudaMemAdvise patch: merged to master, or still downstream?

### Definitive answer: NOT merged. Still a one-off patch attached to discussion #18005.

- The patch commit `c6f6e4f96a7f7bce49f5c21d19ee69fb8b72f84d` (2025-12-14) was
  shared by the discussion author (fairydreaming) as an **experimental
  fork**. It is not referenced by any merged PR on `ggml-org/llama.cpp`.
- The discussion thread itself explicitly notes the patch is experimental and
  has known cleanup-hang issues (NVIDIA driver kernel stack traces during
  `Ctrl+C` exit involving NUMA memory operations).
- No follow-up PR title with "cudaMemAdvise" + "preferred location" + "MoE"
  was found in the merged-PR list for late 2025 or 2026.
- Issue **#16197** "GGML_CUDA_ENABLE_UNIFIED_MEMORY Does not work" (opened
  2025-09-23) is **still open and labeled `stale`** as of late May 2026. The
  underlying problem is not fully resolved upstream.

### What IS in master

- PR **#15077** "llama : add --n-cpu-moe option" by slaren, merged
  **2025-08-04** (commit `ec428b0`). This pre-dates discussion #18005 by
  four months. It exposes `--cpu-moe` (all MoE) and `--n-cpu-moe N` (first
  N MoE layers).
- PR **#12934** "CUDA/HIP: Share the same unified memory allocation logic"
  by hjc4869, merged **2025-04-15**. This is what made
  `GGML_CUDA_ENABLE_UNIFIED_MEMORY` a runtime env var instead of compile-time
  `GGML_HIP_UMA`. The PR author IMbackK explicitly noted in the review
  thread: "im not sure how cudaMallocManaged vs cudaMalloc behave on Nvidia
  apus (grace-hopper/ tegra)." So the in-tree code is **untested on GH200**;
  the GH200-specific MoE-on-Grace behavior is what discussion #18005
  characterizes and the experimental patch tries to fix.

### Why the patch beats the merged flag on GH200

`--n-cpu-moe` uses tensor overrides to allocate expert weights with the CPU
backend from the start. That works fine on PCIe systems. On GH200 with
`GGML_CUDA_ENABLE_UNIFIED_MEMORY=1`, the kernel uses cudaMallocManaged for
*all* tensors so it can oversubscribe HBM. The driver's default migration
heuristic moves expert pages into HBM on first touch and evicts them under
pressure — bad for sparse MoE. The cudaMemAdvise patch explicitly calls
`cudaMemAdviseSetPreferredLocation(expert_tensor, cpu_device)` at init,
overriding the heuristic. **You need both**: UM=1 to map Grace LPDDR into the
GPU address space, and the advise call to keep experts pinned there. Without
UM=1, `--n-cpu-moe` alone gives the same effect via the CPU backend (and is
what most non-GH200 users do).

### Bottom line for users today

If you are on GH200 in May 2026 and want the 18.36 t/s number:
1. Build llama.cpp **with the patch from #18005 cherry-picked**, OR
2. Run with `--n-cpu-moe 58` **without** `GGML_CUDA_ENABLE_UNIFIED_MEMORY=1`
   (this routes experts through the CPU backend, similar effect, no UM
   driver risk). Round 1's recommended command implicitly assumed UM=1 with
   `--n-cpu-moe`; that combination is not what produced the 18.36 t/s
   number and may underperform.

**Correction to round 1**: the recommended config

```
GGML_CUDA_ENABLE_UNIFIED_MEMORY=1 \
./llama-server -m deepseek-v3.1-Q4_K_M.gguf -ngl 99 --n-cpu-moe 58 -fa -c 8192
```

is **not validated** by any public bench. The validated config is either
(a) patched binary + UM=1 + no `--n-cpu-moe`, or (b) stock binary + no UM +
`--n-cpu-moe 58`. Pick (b) for stability today.

## Question 3 — Llama-3.3-70B single-stream tg numbers on GH200

### Public llama.cpp number on GH200 for Llama-3.3-70B (any quant): NONE FOUND.

Checked:
- Discussion #18005 (GH200 perf thread) — only MoE models benched.
- Discussion #15013 (general CUDA scoreboard) — uses Llama-2-7B Q4_0 only;
  no 70B entries on any Hopper SKU.
- Discussion #16578 (DGX Spark perf thread) — close relative on aarch64+CUDA
  (GB10, 122 GB unified). No 70B entries; benches focus on GPT-OSS 20B/120B,
  Qwen3 30B/7B, Gemma 3 4B, GLM 4.5 Air.
- LocalLLaMA subreddit search — no indexed posts pairing GH200 with
  llama.cpp Llama-70B single-stream tg.
- Arxiv 2411.00136 (LLM-Inference-Bench, Oct 2024) — paper claims llama.cpp
  numbers for Llama-2-70B, Llama-3-70B, Qwen-2-72B on GH200 but the HTML
  rendering on arxiv is broken; could not extract specific values. PDF
  needed (out of scope here).

### Adjacent numbers worth knowing

- **Lambda + Baseten, TRT-LLM + SGLang bench, FP8 Llama-3.3-70B on GH200**:
  GH200 is 32% faster than H100 on ShareGPT trace at batch 32. No tokens/sec
  number given. (FP8 path — flagged broken on our DiT stack, but for LLM TRT-LLM
  it works.)
- **Lambda blog, vLLM unquantized Llama-3.1-70B on GH200 vs H100**: GH200
  4.33 t/s vs H100 0.57 t/s (single-stream, max-seq 4096, TP=1). The H100
  number is heavily memory-pressure-bound (70 B unquantized doesn't fit
  comfortably in 80 GB at long context); the 7.6x ratio is dramatic but
  not a steady-state TG comparison.
- **Forum reports on DGX Spark / GB10 (aarch64+CUDA, same family)**: Llama
  3.3 70B FP8 ran at 2-3 t/s — slow because that's a unified-memory machine
  pretending to be a GPU; not directly comparable to GH200 which has real
  HBM3e.

### Order-of-magnitude expectation on GH200 (calculated, not measured)

For a 70 B model at Q4_K_M (~40 GB weights), single-stream decode on a
GH200 with HBM3e ~4.9 TB/s on the 96 GB SKU (or ~4.0 TB/s on 144 GB SKU)
is purely memory-bandwidth-bound. At 4 TB/s effective and 40 GB/token-step,
the ceiling is ~100 t/s; llama.cpp typically reaches **35-55%** of bandwidth
ceiling on dense 70 B Q4, so expect **35-55 t/s** TG single-stream. Round 1
quoted "30-50 t/s" which is consistent.

For Q6_K (~55 GB), bandwidth ceiling ~73 t/s, expect ~25-40 t/s.
For Q8_0 (~74 GB), bandwidth ceiling ~54 t/s, expect ~18-30 t/s.

**These are derived from bandwidth, not measured.** Round 2 / round 3
flagged this gap; nobody has published the direct number as of May 2026.

## Question 4 — MTP speculative decoding (PR #22673) status

### Status: MERGED 2026-05-16.

PR #22673 "llama + spec: MTP Support" by am17an. Verified merged 2026-05-16
(nine days before today's date). Now in master.

### Flags

- `--spec-type draft-mtp` (NOT `--spec-type mtp` as the substack blog wrote;
  the PR uses `draft-mtp`)
- `--spec-draft-n-max N` — max draft tokens per step; 2-3 recommended
- `--spec-draft-ngl N` — draft model GPU layer count (separate from target)

### Supported models

- Qwen3.6-27B-MTP-GGUF
- Qwen3.6-35B-A3B-MTP-GGUF (an MoE target)
- Generally any model with a published MTP head GGUF; works for dense and
  MoE targets.

### Known limitations

- `n_parallel=1` only — speculative + parallel batching is not unified
  in this PR.
- Prefill (PP) takes a **negative hit** because of device-to-host embedding
  transfers in the draft path.
- Not yet supported in `llama-bench` — feature request open as issue #22947.

### Reported benchmarks (NONE on GH200)

- Qwen3.6-27B Q6_K on RTX 3090: 22.97 → 42.45 t/s (~1.85x).
- Qwen3.6 27B on RunPod RTX 3090: ~38 → ~65 t/s (~1.7x).
- Tested also on RTX 3060, Strix Halo, R9700, Vulkan backends.
- **No Grace Hopper / GH200 / aarch64+CUDA numbers in the PR thread.**

### Adjacent: April 2026 HackMD post

Independent test on Qwen3.6-35B-A3B (MoE target) + RTX 3090: **none** of the
spec-decode modes (ngram-cache, ngram-mod, model-draft, DFlash) beat baseline.
Hypothesis attributed to "expert-saturation" on small-active MoE — verification
overhead exceeds savings when the target already runs only ~3 B active params.
This is a real caution for the GH200 MoE story: spec-decode helps dense
targets a lot, may **hurt** small-active MoE targets.

### Implication for GH200

For a dense 70B target with a small dense draft (e.g. Llama-3.2-3B drafting
Llama-3.3-70B Q4_K_M), MTP-style spec decode should yield ~1.5-2x decode
improvement on GH200 — both models fit in HBM, no offload needed, NVLink-C2C
not a factor. **For DeepSeek V3.1 with MoE offload + MTP**, the value is
unclear and unmeasured: the target is already CPU-bound on expert fetch, and
the draft would need to be either (a) all-HBM (different small model with
matching tokenizer) or (b) also MoE-offloaded (probably a wash).

## Question 5 — Diffusion via stable-diffusion.cpp on GH200 (Flux / Wan)

### Direct GH200 / Grace Hopper benchmarks for stable-diffusion.cpp: NONE.

The leejet/stable-diffusion.cpp repo, releases page, and discussions contain
**no** GH200, Grace Hopper, or aarch64+CUDA benchmark numbers as of May 2026.

### What's supported (May 2026 master)

leejet/stable-diffusion.cpp supports:
- SD 1.x, 2.x, 3, 3.5
- Flux
- **Wan 2.1 / 2.2** (T2V and I2V) — yes, the same Wan stack you're working with
- Qwen-Image
- Z-Image

Backends: CPU, CUDA, Metal, Vulkan, SYCL, OpenCL, MUSA, ggml-backend.

### Offload mechanisms (relevant to Grace + HBM split)

- `--offload-to-cpu`: cache model weights in RAM, copy into VRAM on demand.
  Useful for swapping between high/low-noise DiT during Wan pipeline.
- `--clip-on-cpu`: text encoder permanently on CPU.
- `--vae-on-cpu`: VAE permanently on CPU.
- `--control-net-cpu`: ControlNet on CPU.
- `--high-noise-diffusion-model` + standard `--diffusion-model`: Wan-style
  two-stage DiT, can sequentialize the load.

The offload mechanism uses the underlying ggml backend. **It does NOT use
`cudaMallocManaged` / unified memory.** It does explicit
`cudaMemcpyHostToDevice` per swap.

### Would Grace + NVLink-C2C help Wan?

**Theoretical answer: yes, substantially.** Wan 2.2 14B I2V VRAM profile
(from discussion #1129):
- T5 encoder: ~6 GB
- High-noise DiT (14 B at bf16): ~28 GB; quantized Q8 ~15 GB
- Low-noise DiT: same as high-noise
- VAE peak: ~5.4 GB (significant for 720p video)
- Working tensors / latents: a few GB more
- **Peak concurrent allocation: 34+ GB on a 24 GB consumer card → OOM.**

On GH200 with 96 GB HBM3e you can fit the whole pipeline in HBM without
offloading at all. On the older 80 GB H100 you'd need offload; on GH200 you
don't. **So the Grace memory doesn't actually unlock anything Wan-specific
that HBM alone wouldn't.**

Where Grace helps:
1. If you batch many sequences / very long latents (e.g. 5-second 720p video
   needing >40 GB just in activations), the spare LPDDR5X can hold encoder
   buffers / VAE intermediates without DiT eviction.
2. If you run the full Wan 2.2 14B at bf16 (no quantization, no offload) +
   prompt enhancer LLM + safety classifier in the same process, the 480 GB
   pool means no pressure.
3. `--offload-to-cpu` on Grace is much cheaper than on PCIe (135 GiB/s
   measured async memcpy vs ~25 GiB/s on PCIe Gen5 x16). So when Wan
   weight-swap between high-noise and low-noise DiT, the swap is ~5x faster
   on GH200 than on PCIe.

Where Grace does **not** automatically help:
1. stable-diffusion.cpp does not have a `GGML_CUDA_ENABLE_UNIFIED_MEMORY`
   code path. The MoE-style "weights live in CPU and GPU reads them over
   C2C" pattern doesn't exist for DiT (one expert per token doesn't apply).
2. Diffusion is compute-bound (lots of matmul on a single large block of
   weights per step) not bandwidth-bound the way LLM decode is. Grace
   bandwidth doesn't change diffusion steps/sec.
3. **FP8 is broken on this Wan/DiT stack per our standing constraint.**
   stable-diffusion.cpp uses GGUF quants (Q8_0, Q6_K, Q5_K_M, Q4_K_M) which
   are fine; nobody is reporting a working fp8 DiT path here.

### Recommendation

For Wan 2.2 14B I2V on GH200:
- Build stable-diffusion.cpp aarch64+CUDA with `CMAKE_CUDA_ARCHITECTURES=90`.
- Run **without** any `--*-on-cpu` flag — fit everything in 96 GB HBM3e.
- Use Q8_0 weights for quality, Q4_K_M for speed; both fit easily.
- No measured GH200 number exists; this is a research opportunity, not a
  replication.

For Flux on GH200: trivially fits, no offload needed, no Grace benefit
beyond "you don't have to think about VRAM."
