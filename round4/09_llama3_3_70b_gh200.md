# Llama-3.3-70B-Instruct on a Single GH200 — Definitive Recipe + Benchmark Plan

**Round 4, deep dive #09**
**Date:** 2026-05-25
**Target:** 1× NVIDIA GH200 Grace-Hopper Superchip (96 GB HBM3 + 480 GB LPDDR5X over NVLink-C2C 900 GB/s)
**Hard prior (carried):** FP8 is broken on this stack — bf16 is the default precision for every command in this document. FP8 callouts are noted but never the recommendation.

This document collects every published tok/s number for Llama-3.3-70B (and adjacent 70B variants) on Hopper-class hardware, triangulates an expected envelope for vLLM 0.21 V1 EAGLE-3 BF16 on a single GH200, and ends with the exact vLLM and TensorRT-LLM commands plus a one-hour benchmark script.

### 1d. Lambda × Baseten "Partner Spotlight" — Llama 3.3 70B FP8 on GH200 (Feb 2025)

The most-cited GH200/H100 head-to-head. Now read **exactly** for what it does and doesn't publish:

- Engine: **TensorRT-LLM** (Baseten Engine Builder), benchmarked with **SGLang's bench harness** on ShareGPT
- Precision: **FP8** (model footprint ~70 GB, leaving 27 GB on GH200-96 vs 10 GB on H100-80)
- Batch size: **32**
- Hardware: 1× GH200-96GB vs 1× H100-80GB
- Headline: **"GH200 outperformed H100 by 32%"** on aggregate throughput
- Sequence lengths: ShareGPT — variable, no exact ISL/OSL published
- **No absolute tok/s number was ever published.** Both blog posts only state the relative 32% delta.

The 32% delta is decomposed in the post itself: GH200 has 21% higher HBM bandwidth than H100 (4.0 vs 3.35 TB/s) **and identical compute** — so the remaining ~11 percentage points come from KV-cache headroom (27 GB vs 10 GB) plus the C2C-backed CPU-offload, **not from raw compute**. The number is real, the engine version (TRT-LLM, ~early-2025) is irrelevant for vLLM extrapolation, and **anyone quoting an absolute tok/s for "GH200 Llama-3.3-70B" derived from this post is making it up**.

### 1e. Lambda Labs 2024-11-22 blog — Llama 3.1 70B BF16 on GH200 vs H100

This is the one **published absolute number** that puts GH200 head-to-head with H100 on BF16:

- Model: Llama 3.1 70B (**note: 3.1, not 3.3**), **no quantization** ("no quant" — implied BF16; FP32 unlikely for 140 GB+ on 80 GB / 96 GB)
- Framework: vLLM (pre-V1, ~vLLM 0.6 era)
- max_seq_len: 4,096
- TP=1
- CPU offload: GH200 = **60 GB**, H100 SXM = **75 GB**
- num_tokens: 1,024 (output)
- Result: **GH200 = 4.33 tok/s; H100 SXM = 0.57 tok/s → 7.6×**
- Cost: GH200 $0.02/Mtok vs H100 $0.16/Mtok → ~8× cheaper

The benchmark repo (`LambdaLabsML/vllm-gh200-vs-h100`) does NOT publish vLLM version, dtype flag, max-num-seqs, batch-size, or gpu-memory-utilization. The 4.33 tok/s number is a single-stream / very-low-concurrency result on legacy vLLM — V0 engine, ~0.6.x. The 7.6× delta is the H100 being absolutely crippled by swapping its 75 GB CPU offload over PCIe Gen5 while GH200 swaps 60 GB over the 900 GB/s C2C link. **Treat this as a lower-bound floor that V1 has long since beaten, not a representative number.**

## 2. Baseten/Lambda Feb-2025 "+32%" — exact config

Cross-confirmed across both blog posts. Full configuration:

- **Hardware:** 1× GH200 96GB vs 1× H100 80GB SXM (Lambda Cloud)
- **Model:** Llama-3.3-70B in **FP8** (~70 GB on disk)
- **Engine:** TensorRT-LLM via Baseten Engine Builder (proprietary build pipeline wrapping TRT-LLM; the underlying TRT-LLM version is unstated but post date 2025-02-07 → ≈v0.15-0.17)
- **Batch size:** 32 concurrent requests
- **Benchmark harness:** SGLang's `bench_serving` with built-in KV-cache-reuse test
- **Dataset:** ShareGPT (variable ISL/OSL — distribution not published)
- **TP:** 1 (single GPU per config)
- **Result:** GH200 +32% aggregate throughput vs H100 — **no absolute tok/s published in either post**
- **Decomposition:** 21% from 4.0 vs 3.35 TB/s HBM bandwidth + ~11% from KV-cache headroom (27 GB free vs 10 GB free after weights)

**The number is methodologically sound but not reproducible at absolute scale because no absolute tok/s, vLLM version, KV-offload size, or ISL/OSL was ever published.** Any future "GH200 is X× faster than H100 on Llama-3.3-70B" claim that doesn't show the same configuration is conflating Baseten's relative delta with a number it never produced.

## 3. Triangulated expected tok/s envelope for vLLM 0.21 V1 BF16 EAGLE-3, single GH200

This is an estimate, not a measurement. Methodology: anchor on three published reference points and bound the answer between them.

### 3a. Anchors

1. **NVIDIA H200 TRT-LLM FP8 batch=1 baseline (no spec): 51.14 tok/s/user** (Llama-3.3-70B, single H200, TP=1, FP8). Source: NVIDIA blog 2024-12-11.
2. **NVIDIA H200 TRT-LLM FP8 with Llama-3.2-1B draft: 181.74 tok/s/user** (3.55× speedup, same blog).
3. **NVIDIA NIM 2×H100 80GB FP8 TP=2 Llama-3.3-70B @ concurrency 100, 500/2000 ISL/OSL: 3,322.87 tok/s aggregate**, single H100 share ≈1,660 tok/s. Source: NIM benchmarks.
4. **morphllm 1×H100 80GB FP8 vLLM batch=64: 460 tok/s aggregate** for Llama-3.1-70B FP8. Source: morphllm.
5. **Baseten +32% GH200 over H100 FP8 batch=32** (relative only).

### 3b. Adjustments

- **BF16 vs FP8 throughput penalty (decode-heavy, batch≥16):** roughly 1.4-1.7× slower for memory-bandwidth-bound 70B decode. At batch=1 the penalty is smaller (~1.2-1.4×) because both are bandwidth-limited. (vLLM blog 2026-04-22 measured 14.9% FP8 advantage at heavy load on Llama-3.1-8B — smaller model, less BW-bound, so 70B BW penalty is larger.)
- **GH200 vs H100 same-precision adjustment:** +20-32% from HBM3 (4.0 TB/s vs 3.35 TB/s = +19%) + KV headroom (+11% Baseten). Both fold into one factor: **×1.2-1.32**.
- **vLLM V1 vs V0 throughput uplift:** ~**1.7×** average (vLLM blog), more on V1's KV-offload + chunked-prefill path. Most modern published H100 numbers already are V1.
- **EAGLE-3 uplift on 70B:**
  - Single-user (batch=1): 1.6-2.6× (Red Hat 2025-07; NVIDIA TRT-LLM published 3.55× with 1B draft — that's TRT-LLM not vLLM, and TRT-LLM's spec-decode kernel pipeline is more aggressive)
  - Concurrency 8-200: 9.5-20.7% on output throughput (Red Hat 2026-04-16, gpt-oss benchmark, persists to 200 concurrent)
- **TP=1 single-die advantage:** No cross-die communication on a single GH200, so unlike the 70B-on-2×H200 forum report (40 tok/s/user, likely NCCL-overhead-bound), TP=1 on one die avoids that tax.

### 3c. Envelope estimates (BF16, vLLM 0.21 V1, EAGLE-3 `num_speculative_tokens=3`, single GH200, 96 GB HBM)

Confidence: **medium-low** (±30%). These are interpolations, not measurements.

| Concurrency | Workload shape (ISL → OSL) | No spec-decode (tok/s/user) | With EAGLE-3 (tok/s/user) | Aggregate w/ EAGLE-3 (tok/s) | Confidence |
|---|---|---|---|---|---|
| **1** | 500 → 2000 | 35-45 | **60-110** | 60-110 | M |
| **16** | 500 → 2000 | 25-35/user | 35-50/user (1.4× tail) | **560-800** | M |
| **64** | 1000 → 1000 | 12-18/user | 14-22/user (10-25% lift at concurrency) | **900-1,400** | M-L |
| **256** | 500 → 2000 | 5-9/user | 6-11/user (8-15% lift) | **1,500-2,800** | L |

Notes on the table:
- The **batch=1** estimate of 35-45 tok/s/user (no spec) is grounded in H200's published 51 tok/s for FP8 batch=1. BF16 is ~1.2× slower at batch=1 → 35-42; GH200's HBM advantage is small at batch=1 → +0-5%. EAGLE-3 with 1B draft can hit 3.55× on TRT-LLM but vLLM's spec-decode kernel is less polished — assume 1.5-2.5× — yielding 60-110.
- The **batch=64** aggregate of 900-1,400 tok/s in BF16 maps to roughly half of what the morphllm 460 tok/s number suggests scaled — 460 was a single-H100 FP8 batch-64 → BF16 1.5× slower → 300, then GH200 +25% = 375 single-stream. But the table is for *aggregate* with concurrency 64. Aggregate ≈ per-user × users, and per-user falls with concurrency. The number 900-1,400 tracks NIM's 4×H100 BF16 sweep at 250 concurrent of 2,600 tok/s aggregate divided by 4 GPUs = 650 tok/s/GPU, scaled up by GH200/H100 ratio 1.25× and EAGLE-3 lift 1.15× → ~930. Upper bound assumes a more decode-heavy shape and higher KV headroom utilization.
- The **batch=256** estimate of 1,500-2,800 is the most uncertain because: (a) GH200's 96 GB HBM caps how many KV slots fit even with offload, (b) EAGLE-3 lift narrows further at high concurrency. Lower bound assumes prefill-bound saturation, upper assumes KV-offload to Grace LPDDR delivers full C2C bandwidth.

### 3d. What would shift these numbers

- **Better than +30%:** if Grace-LPDDR KV offload over C2C truly delivers full 900 GB/s for KV traffic, the batch=256 number could trend up to 3-4k tok/s. No published benchmark proves this on real workloads.
- **Worse than -30%:** if vLLM 0.21's HMA path has any GH200-specific regression (unverified — no GH200 benchmark exists for 0.21), numbers could collapse. The vLLM forum report of 40 tok/s/user on 2×H200 for Llama-3.3-70B BF16 suggests TP-related overhead can dominate; TP=1 on GH200 should avoid that.
- **Worse if you forget CPU offload:** with bf16 weights ~140 GB on a 96 GB GPU you *cannot* run without `--cpu-offload-gb`. Forgetting it means OOM at startup. With it set to 60-80 GB on Grace LPDDR, you have ~80 GB HBM free for KV + activations.

## 4. Exact `vllm serve` command for max throughput on single GH200

Pin vLLM 0.21.0 (2026-05-15, latest stable as of 2026-05-25). C++20 compiler required, Transformers v5 required, CUDA 13.0+ recommended.

### 4a. Throughput-mode launch (recommended starting point)

```bash
# NUMA pin: bind to the single GPU's NUMA domain
# GH200 exposes 2 NUMA nodes: 0 = Grace LPDDR (CPU-local), 1 = HBM (GPU-local).
# Pin the worker process to node 0 for CPU; vLLM allocates GPU memory directly.
# Use --numa-bind so vLLM's spawn pre-pins the interpreter.

numactl --cpunodebind=0 --membind=0 \
  vllm serve meta-llama/Llama-3.3-70B-Instruct \
    --tensor-parallel-size 1 \
    --dtype bfloat16 \
    --max-model-len 32768 \
    --gpu-memory-utilization 0.92 \
    --enable-prefix-caching \
    --enable-chunked-prefill \
    --max-num-batched-tokens 8192 \
    --max-num-seqs 256 \
    --cpu-offload-gb 60 \
    --swap-space 64 \
    --kv-cache-dtype auto \
    --async-scheduling \
    --numa-bind \
    --speculative-config '{"method":"eagle3","model":"yuhuili/EAGLE3-LLaMA3.3-Instruct-70B","num_speculative_tokens":3,"draft_tensor_parallel_size":1}' \
    --host 0.0.0.0 --port 8000 \
    --served-model-name llama-3.3-70b
```

Per-flag rationale:

| Flag | Value | Why |
|---|---|---|
| `--tensor-parallel-size` | **1** | Single Hopper die. Don't shard. |
| `--dtype` | **bfloat16** | Hard prior. FP8 has the head_dim caveat (Llama is head_dim=128 so safe in principle, but bf16 is the constraint). |
| `--max-model-len` | **32768** | Throughput-friendly cap. Bump to 131072 only if you need long context — see §6. |
| `--gpu-memory-utilization` | **0.92** | Aggressive but safe with offload available as overflow. |
| `--enable-prefix-caching` | **on** | V1 default; <1% throughput hit at 0% hit rate, multiplies at high hit. |
| `--enable-chunked-prefill` | **on** | V1 default. Required for balanced TTFT vs ITL. |
| `--max-num-batched-tokens` | **8192** | The vLLM Hopper recipe's recommended value. Higher = better TTFT; lower (2048) = better ITL. |
| `--max-num-seqs` | **256** | GH200's HBM + LPDDR KV headroom supports it. Most H100 deployments stop at 64-128. |
| `--cpu-offload-gb` | **60** | Lambda's 2024 GH200 number used 60 GB. Spills weights into Grace LPDDR over C2C; frees ~60 GB HBM for KV. Bump to 80 if you need more KV. |
| `--swap-space` | **64** | CPU swap for preempted KV blocks; Grace LPDDR makes this nearly free. |
| `--kv-cache-dtype` | **auto** (= bf16) | Per hard prior. **Do not** set `fp8_e4m3` blindly. |
| `--async-scheduling` | **on** | Zero-bubble overlap between scheduling and GPU exec. Default in 0.21. |
| `--numa-bind` | **on** | Auto-detects GPU-to-NUMA mapping; uses `numactl --cpunodebind --membind` per worker so the interpreter, imports, and early allocator state are pinned. |
| `--speculative-config` | EAGLE-3 head, 3 draft tokens | Pretrained head from `yuhuili/EAGLE3-LLaMA3.3-Instruct-70B`. Sweet spot is 2-3 per Red Hat 2026-04-16. |

### 4b. Long-context launch (128k window, KV-offload backend)

```bash
numactl --cpunodebind=0 --membind=0 \
  vllm serve meta-llama/Llama-3.3-70B-Instruct \
    --tensor-parallel-size 1 \
    --dtype bfloat16 \
    --max-model-len 131072 \
    --gpu-memory-utilization 0.90 \
    --enable-prefix-caching \
    --enable-chunked-prefill \
    --max-num-batched-tokens 16384 \
    --max-num-seqs 128 \
    --cpu-offload-gb 80 \
    --kv_offloading_backend native \
    --kv_offloading_size 256 \
    --swap-space 64 \
    --async-scheduling \
    --numa-bind \
    --host 0.0.0.0 --port 8000
```

Drops EAGLE-3 because spec decode loses at very long context (draft model context window cap is 2048 in the current vLLM speculators stack as of v0.9.1 per Red Hat). Bumps KV-offload size to 256 GB (well under the documented OOM-at->1024 GB ceiling). Drops batch to 128 because 128k KV per request is large enough that 256 sequences would saturate even Grace LPDDR.

## 6. KV-cache memory math — bf16 weights, headroom at 32k and 128k

### 6a. Model footprint

- Llama-3.3-70B-Instruct in bf16: **~140 GB** weights (≈70B × 2 bytes).
- GH200 has **96 GB HBM** + **480 GB LPDDR5X** (Grace) over **900 GB/s** C2C.
- 140 > 96, so on a single GH200 you must offload weights. `--cpu-offload-gb 60` → ~60 GB of weights in LPDDR, 80 GB in HBM. `--cpu-offload-gb 80` → 80 GB LPDDR, 60 GB HBM.

### 6b. KV-cache footprint per token (Llama-3.3-70B in bf16)

Llama-3.3-70B uses GQA: 80 layers, 8 KV heads, head_dim 128. Per token:
- KV = 2 (K+V) × 80 layers × 8 KV heads × 128 dim × 2 bytes (bf16)
- = **327,680 bytes/token = 320 KB/token**

(Cross-checked with public Llama 3.1/3.3 70B specs — same architecture. Lyceum's value of "0.31 MB per token" rounds to the same 320 KB.)

Or equivalently, 1k tokens of KV = **320 MB** in bf16.

### 6c. Headroom math

After loading weights with `--cpu-offload-gb 60` and `--gpu-memory-utilization 0.92`:

- Total HBM: 96 GB
- vLLM reserves: 0.92 × 96 = **88.3 GB usable**
- Weights in HBM: 140 - 60 = **80 GB**
- Activations + CUDA workspace: ~**4-6 GB**
- **HBM KV budget: 88.3 - 80 - 5 ≈ 3.3 GB** ← only **~10k tokens of KV** fit on-die

This is tight. Three escape valves:

1. **Push more weights to LPDDR.** With `--cpu-offload-gb 80`: HBM KV budget = 88 - 60 - 5 = **23 GB ≈ 71k tokens KV.** This is the practical setting.
2. **Use `--kv_offloading_backend native --kv_offloading_size 256`** to dedicate 256 GB of Grace LPDDR explicitly to KV overflow. With this enabled, vLLM 0.21's HMA can dynamically migrate KV blocks between HBM (hot) and LPDDR (warm). Active KV stays in HBM, history KV spills via C2C. **No hard cap on practical context length** below the 1024 GB OOM threshold.
3. **`--kv-cache-dtype fp8_e4m3`** halves KV size. Per the FP8 caveat, the two-level FP32 accumulation fix is in vLLM 0.19.1+ so accuracy is recoverable for Llama (head_dim=128, not 256). **Still NOT the default per the hard prior.** Only use after a 128k-needle sanity check.

### 6d. KV demand at common contexts

How much KV memory do you actually need given concurrency and context?

| Context per request | KV per request (bf16) | Concurrency 16 | Concurrency 64 | Concurrency 256 |
|---|---|---|---|---|
| 2k (chat) | 640 MB | 10 GB | 41 GB | 164 GB |
| 8k | 2.56 GB | 41 GB | 164 GB | **655 GB (overflows LPDDR)** |
| 32k | 10.2 GB | **164 GB (LPDDR)** | **655 GB (overflows LPDDR)** | impossible |
| 128k | **41 GB** | 655 GB (overflows LPDDR) | impossible | impossible |

Reading the table: at 32k context with 16 concurrent users you need 164 GB of KV — fits in Grace's 480 GB LPDDR (set `--kv_offloading_size 200`), does **not** fit on HBM alone. At 128k with even one user you need 41 GB of KV, near the entire HBM budget after weights. **Single GH200 with 128k context and concurrency > 1 requires the KV-offload backend mandatorily.**

This is also why the Lambda 2024 4.33 tok/s number was so low — at max_seq_len=4096 with the legacy V0 engine doing inefficient SWAP-based offload over a Hopper-era code path that didn't fully exploit C2C, every request was thrashing. V1 + native KV-offload should be dramatically better.

### 6e. With FP8 KV (NOT the default, just for reference)

FP8 KV halves the per-token footprint to **160 KB/token** so each row in the table above halves. Concurrency 64 at 32k drops from 655 GB → 327 GB — still needs LPDDR overflow but now fits more comfortably. The vLLM 0.21 path with the two-level accumulation fix recovers 97-98% of bf16's 128k-needle AUC per the 2026-04-22 blog. **Mentioned for completeness; bf16 is the policy.**

## 7. Long-context (128k needle-in-haystack) — known regression points

### 7a. The 128k FP8 collapse and its fix

The reference data point everyone should know:

- **128k needle-in-a-haystack accuracy on Llama-3.3-70B in BF16 (baseline): 91%**
- Same test, FP8 Flash-Attention-3 on Hopper with FP32 accumulation imprecision: **13%** (catastrophic)
- After two-level FP32 accumulation fix (flash-attention #104, in vLLM 0.19.1+): **89%**

Source: vLLM blog 2026-04-22 "State of FP8 KV-Cache and Attention Quantization."

For Llama-3.3-70B specifically (head_dim=128), the fix is clean: prefill is competitive with bf16, decode is faster, accuracy AUC@128k recovers to 97-98% of bf16. **For head_dim=256 models (Gemma-4-E2B), the fix introduces a 1.6× prefill slowdown that hasn't been resolved as of vLLM 0.21.0** — irrelevant for Llama but watch for it on Gemma deployments.

### 7b. Other long-context regression points

Empirically observed on Hopper at 128k context:

1. **Async scheduler bug** on NGC vllm 26.01-py3 at long context (CUDA-graph crashes / warp exceptions). Fixed in 26.04+. Mitigation: `--enforce-eager` or upgrade.
2. **Spec decode at long context** loses its win — the draft model's context window cap is 2048 in current vLLM speculators stack (as of v0.9.1 baseline). EAGLE-3 will under-accept on a 100k prompt because the draft can only see the last 2k. Disable spec decode when serving 128k workloads.
3. **Prefix-cache hit rate falls** with very long unique prefixes. Keep `--enable-prefix-caching` on because the cost of misses is ~1%, but don't expect free wins on long unique-context workloads.
4. **GH200's 96 GB HBM** caps simultaneous 128k requests to ~1-2 even with KV-offload backend, because the active page set still needs HBM residence. KV-offload won't save you from a hot-set bigger than HBM minus weights minus activations.
5. **`--kv_offloading_size > 1024 GB`** OOM bug (vLLM 0.15 issue #36623, closed without explicit fix note). Practical ceiling on GH200: stay ≤ 400 GB.
6. **`max-num-batched-tokens` interaction** with chunked prefill: only the first chunk hits the prefix cache. For very long prompts (>16k unique tokens), bump `--max-num-batched-tokens` to 16384 or 32768 so the prompt fits in fewer chunks; otherwise prefix cache misses dominate.

## 9. One-hour benchmark script

Drops into a directory and produces a CSV with tok/s for both vLLM and TRT-LLM across batch=1/16/64/256 for BF16 with and without EAGLE-3. Designed to fit in ~60 min wall clock on a single GH200.

```bash
#!/usr/bin/env bash
# /home/ubuntu/research/gh200_inference/round4/bench_70b_gh200.sh
# 1-hour Llama-3.3-70B GH200 benchmark.
# Two engines × two spec-decode modes × four concurrency points = 16 measurements.
# Each measurement runs c×5 requests of 1024→1024 random tokens with ignore_eos.

set -euo pipefail
RESULTS=/tmp/gh200_70b_bench_$(date +%Y%m%d_%H%M%S)
mkdir -p "$RESULTS"
echo "engine,spec,concurrency,output_tok_s,ttft_p50_ms,itl_p50_ms,e2e_p50_ms" > "$RESULTS/results.csv"

MODEL=meta-llama/Llama-3.3-70B-Instruct
EAGLE_VLLM=yuhuili/EAGLE3-LLaMA3.3-Instruct-70B
EAGLE_TRT=nvidia/Llama-3.3-70B-Instruct-Eagle3
CONC_LIST="1 16 64 256"
ISL=1024; OSL=1024

# ============================================================
# Phase 1: vLLM 0.21 — no spec decode
# ============================================================
echo "=== Phase 1: vLLM 0.21 bf16, no spec ==="
numactl --cpunodebind=0 --membind=0 vllm serve "$MODEL" \
  --tensor-parallel-size 1 --dtype bfloat16 \
  --max-model-len 8192 --gpu-memory-utilization 0.92 \
  --enable-prefix-caching --max-num-batched-tokens 8192 \
  --max-num-seqs 256 --cpu-offload-gb 60 --swap-space 64 \
  --async-scheduling --numa-bind \
  --host 0.0.0.0 --port 8000 > "$RESULTS/vllm_noeagle.log" 2>&1 &
VLLM_PID=$!
# Wait for /health to come up (max 10 min for 70B + offload warmup)
until curl -sf http://localhost:8000/health > /dev/null; do sleep 5; done
echo "vLLM ready"

for c in $CONC_LIST; do
  python -m vllm.entrypoints.cli.benchmark \
    --backend openai \
    --model "$MODEL" \
    --base-url http://localhost:8000 \
    --dataset-name random \
    --random-input-len $ISL --random-output-len $OSL --random-prefix-len 0 \
    --num-prompts $((c*5)) --max-concurrency $c \
    --ignore-eos \
    --save-result --result-dir "$RESULTS" \
    --result-filename "vllm_noeagle_c${c}.json" \
    --percentile-metrics ttft,tpot,itl,e2el
  TPS=$(jq '.output_throughput' "$RESULTS/vllm_noeagle_c${c}.json")
  TTFT=$(jq '.median_ttft_ms' "$RESULTS/vllm_noeagle_c${c}.json")
  ITL=$(jq '.median_itl_ms' "$RESULTS/vllm_noeagle_c${c}.json")
  E2E=$(jq '.median_e2el_ms' "$RESULTS/vllm_noeagle_c${c}.json")
  echo "vllm,none,$c,$TPS,$TTFT,$ITL,$E2E" >> "$RESULTS/results.csv"
done

kill $VLLM_PID; wait $VLLM_PID 2>/dev/null || true
sleep 30

# ============================================================
# Phase 2: vLLM 0.21 — EAGLE-3 spec decode
# ============================================================
echo "=== Phase 2: vLLM 0.21 bf16 + EAGLE-3 ==="
numactl --cpunodebind=0 --membind=0 vllm serve "$MODEL" \
  --tensor-parallel-size 1 --dtype bfloat16 \
  --max-model-len 8192 --gpu-memory-utilization 0.90 \
  --enable-prefix-caching --max-num-batched-tokens 8192 \
  --max-num-seqs 256 --cpu-offload-gb 60 --swap-space 64 \
  --async-scheduling --numa-bind \
  --speculative-config "{\"method\":\"eagle3\",\"model\":\"$EAGLE_VLLM\",\"num_speculative_tokens\":3,\"draft_tensor_parallel_size\":1}" \
  --host 0.0.0.0 --port 8000 > "$RESULTS/vllm_eagle.log" 2>&1 &
VLLM_PID=$!
until curl -sf http://localhost:8000/health > /dev/null; do sleep 5; done

for c in $CONC_LIST; do
  python -m vllm.entrypoints.cli.benchmark \
    --backend openai --model "$MODEL" \
    --base-url http://localhost:8000 \
    --dataset-name random \
    --random-input-len $ISL --random-output-len $OSL --random-prefix-len 0 \
    --num-prompts $((c*5)) --max-concurrency $c --ignore-eos \
    --save-result --result-dir "$RESULTS" \
    --result-filename "vllm_eagle_c${c}.json" \
    --percentile-metrics ttft,tpot,itl,e2el
  TPS=$(jq '.output_throughput' "$RESULTS/vllm_eagle_c${c}.json")
  TTFT=$(jq '.median_ttft_ms' "$RESULTS/vllm_eagle_c${c}.json")
  ITL=$(jq '.median_itl_ms' "$RESULTS/vllm_eagle_c${c}.json")
  E2E=$(jq '.median_e2el_ms' "$RESULTS/vllm_eagle_c${c}.json")
  echo "vllm,eagle3,$c,$TPS,$TTFT,$ITL,$E2E" >> "$RESULTS/results.csv"
done

kill $VLLM_PID; wait $VLLM_PID 2>/dev/null || true
sleep 30

# ============================================================
# Phase 3: TRT-LLM 1.2.1 PyTorch backend — no spec decode
# ============================================================
echo "=== Phase 3: TRT-LLM 1.2.1 PyTorch bf16, no spec ==="
cat > /tmp/llama33-bf16.yml <<EOF
enable_attention_dp: false
print_iter_log: false
cuda_graph_config:
  enable_padding: true
  max_batch_size: 256
kv_cache_config:
  free_gpu_memory_fraction: 0.85
  host_cache_size: 200000000000
  enable_block_reuse: true
EOF

# Inside the trtllm container (assumed already running, this script run inside it)
# If you need to launch: docker run ... nvcr.io/nvidia/tensorrt-llm/release:1.2.1
numactl --cpunodebind=0 --membind=0 trtllm-serve "$MODEL" \
  --backend pytorch --dtype bfloat16 --tp_size 1 --ep_size 1 \
  --max_batch_size 256 --max_num_tokens 8192 --max_seq_len 8192 \
  --kv_cache_free_gpu_memory_fraction 0.85 \
  --trust_remote_code --config /tmp/llama33-bf16.yml \
  --host 0.0.0.0 --port 8001 > "$RESULTS/trtllm_noeagle.log" 2>&1 &
TRTLLM_PID=$!
until curl -sf http://localhost:8001/v1/models > /dev/null; do sleep 5; done

for c in $CONC_LIST; do
  python -m tensorrt_llm.serve.scripts.benchmark_serving \
    --model "$MODEL" --backend openai \
    --base-url http://localhost:8001 \
    --dataset-name random \
    --random-input-len $ISL --random-output-len $OSL --random-prefix-len 0 \
    --num-prompts $((c*5)) --max-concurrency $c --ignore-eos \
    --save-result --result-dir "$RESULTS" \
    --result-filename "trtllm_noeagle_c${c}.json" \
    --percentile-metrics ttft,tpot,itl,e2el
  TPS=$(jq '.output_throughput' "$RESULTS/trtllm_noeagle_c${c}.json")
  TTFT=$(jq '.median_ttft_ms' "$RESULTS/trtllm_noeagle_c${c}.json")
  ITL=$(jq '.median_itl_ms' "$RESULTS/trtllm_noeagle_c${c}.json")
  E2E=$(jq '.median_e2el_ms' "$RESULTS/trtllm_noeagle_c${c}.json")
  echo "trtllm,none,$c,$TPS,$TTFT,$ITL,$E2E" >> "$RESULTS/results.csv"
done

kill $TRTLLM_PID; wait $TRTLLM_PID 2>/dev/null || true
sleep 30

# ============================================================
# Phase 4: TRT-LLM 1.2.1 PyTorch backend — EAGLE-3 spec decode
# ============================================================
echo "=== Phase 4: TRT-LLM 1.2.1 PyTorch bf16 + EAGLE-3 ==="
numactl --cpunodebind=0 --membind=0 trtllm-serve "$MODEL" \
  --backend pytorch --dtype bfloat16 --tp_size 1 --ep_size 1 \
  --max_batch_size 256 --max_num_tokens 8192 --max_seq_len 8192 \
  --kv_cache_free_gpu_memory_fraction 0.80 \
  --speculative_config "{\"decoding_type\":\"Eagle\",\"speculative_model\":\"$EAGLE_TRT\",\"num_eagle_layers\":3}" \
  --trust_remote_code --config /tmp/llama33-bf16.yml \
  --host 0.0.0.0 --port 8001 > "$RESULTS/trtllm_eagle.log" 2>&1 &
TRTLLM_PID=$!
until curl -sf http://localhost:8001/v1/models > /dev/null; do sleep 5; done

for c in $CONC_LIST; do
  python -m tensorrt_llm.serve.scripts.benchmark_serving \
    --model "$MODEL" --backend openai \
    --base-url http://localhost:8001 \
    --dataset-name random \
    --random-input-len $ISL --random-output-len $OSL --random-prefix-len 0 \
    --num-prompts $((c*5)) --max-concurrency $c --ignore-eos \
    --save-result --result-dir "$RESULTS" \
    --result-filename "trtllm_eagle_c${c}.json" \
    --percentile-metrics ttft,tpot,itl,e2el
  TPS=$(jq '.output_throughput' "$RESULTS/trtllm_eagle_c${c}.json")
  TTFT=$(jq '.median_ttft_ms' "$RESULTS/trtllm_eagle_c${c}.json")
  ITL=$(jq '.median_itl_ms' "$RESULTS/trtllm_eagle_c${c}.json")
  E2E=$(jq '.median_e2el_ms' "$RESULTS/trtllm_eagle_c${c}.json")
  echo "trtllm,eagle3,$c,$TPS,$TTFT,$ITL,$E2E" >> "$RESULTS/results.csv"
done

kill $TRTLLM_PID; wait $TRTLLM_PID 2>/dev/null || true

# ============================================================
# Phase 5: summary
# ============================================================
echo
echo "=== RESULTS ==="
column -ts, "$RESULTS/results.csv"
echo
echo "CSV at: $RESULTS/results.csv"
echo "Logs at: $RESULTS/*.log"
echo "Raw JSON per-run at: $RESULTS/*_c*.json"
```
