# Round 4 Agent #10 — MoE Models on a Single GH200 (Headline Use Case)

**Agent**: #10 / Round 4
**Date**: 2026-05-25
**Scope**: Deep dive on DeepSeek-V3.1 (671B), Kimi K2 Thinking (~1T), GPT-OSS 120B, Mixtral 8x22B, Qwen3-Next-80B-A3B on a single GH200. Recipes, batching, concurrency, active-parameter routing efficiency.
**Constraint carried**: FP8 broken on this GH200/Wan stack — bf16 / Q4_K_M / MXFP4 default.

---

## 0. Why GH200 is the right shape for MoE (one-paragraph thesis)

A single GH200 is **96 GB HBM3 (or 144 GB HBM3e) + 480 GB LPDDR5X + 900 GB/s NVLink-C2C** in one cache-coherent package. That gives **~528–576 GB unified addressable memory** with no PCIe hop. MoE models are characterized by a **sparse active set**: DeepSeek V3.1 671B touches only ~37B params per token (8 of 256 routed experts + 1 shared, per layer); Kimi K2 Thinking ~1T touches ~32B; GPT-OSS 120B touches ~5.1B (top-4 of 128). The entire weight tensor must be resident **somewhere**; the active subset moves through compute each token. GH200 is the only single-package config in May 2026 where a Q3–Q4-quantized 671B or 1T MoE fits **in memory at all** without multi-GPU TP. The Round-1 number — **DeepSeek V3.1 Q4_K_M at 18.30 t/s TG single-stream** — is the headline; this report unpacks how to get it, how concurrency scales, and what the alternatives (vLLM, SGLang) can and cannot do.

## 1. The 18.3 t/s DeepSeek V3.1 number — exact recipe

### 1.1 Source and provenance

llama.cpp discussion **#18005 "Performance of llama.cpp on NVIDIA Grace Hopper GH200 (+optimizations)"** by user `fairydreaming`, opened **2025-12-13**, optimization patch landed **2025-12-14** (commit `c6f6e4f96a7f7bce49f5c21d19ee69fb8b72f84d`).

### 1.2 Hardware under test

- GH200 **144 GB HBM3e** SKU (not the 96 GB)
- Grace Neoverse-V2, 72 cores @ 3.357 GHz
- 480 GB LPDDR5X @ 6400 MT/s (datasheet 546 GB/s; fairydreaming's instrumentation reports ~267 GB/s usable in his test)
- CUDA 13.0

### 1.3 Model file

- **DeepSeek-V3.1 Q4_K_M GGUF**
- File size: **377.55 GiB**, parameter count 671.03 B
- This is the standard `unsloth/DeepSeek-V3.1-GGUF` Q4_K_M variant (`-ot ".ffn_.*_exps.=CPU"`-compatible layout)
- Unsloth also ships UD-Q4_K_XL at 384 GB and UD-Q2_K_XL at 251 GB if HBM+LPDDR is tighter

### 1.4 The benchmark command

The headline single-stream numbers come from `llama-bench`, not `llama-server`:

```bash
# With the cudaMemAdvise patch baked in, OR equivalently --n-cpu-moe on master:
GGML_CUDA_ENABLE_UNIFIED_MEMORY=1 \
./build/bin/llama-bench \
  -m DeepSeek-V3.1-Q4_K_M.gguf \
  -fa 1 \
  -d 0,4096,8192 \
  -p 2048 \
  -n 32 \
  -ub 2048
```

Result on GH200 144 GB:

| Mode | pp2048 (t/s) | tg32 (t/s) |
|---|---:|---:|
| UM naive (no patch) | 41.63 | 8.86 |
| UM + cudaMemAdvise patch | **609.89 ± 2.68** | **18.30 ± 0.02** |

The patch is a **2.07× speedup on TG** and a **14.6× speedup on PP**. Naive unified memory thrashes expert pages between HBM and LPDDR because CUDA's UVM heuristic doesn't know which tensors are MoE-cold.

### 1.5 The cudaMemAdvise patch — what it actually does

```c
// In ggml-cuda backend init, when allocating each tensor:
bool isHostTensor = strstr(tensor->name, "_exps.weight") != NULL;
cudaMemLocation loc = {
    .type = isHostTensor ? cudaMemLocationTypeHost : cudaMemLocationTypeDevice,
    .id   = ctx->device
};
CUDA_CHECK(cudaMemAdvise(
    (char*)tensor->data, alloc_size,
    cudaMemAdviseSetPreferredLocation, loc));
```

It scans tensor names: anything matching `*_exps.weight` (the routed expert FFN tensors `ffn_up_exps`, `ffn_gate_exps`, `ffn_down_exps`) is **pinned-preferred to host (Grace LPDDR5X)**. Everything else (attention, embeddings, dense FFN, shared expert, norms) gets preferred-on-device (HBM3e). Per-token, activations cross C2C, the expert matmul runs on the GPU **reading the Grace-resident weights over NVLink-C2C at ~375 GB/s** (verda.com host→device STREAM measurement), result returns over the same fabric.

### 1.6 Equivalent modern recipe (replaces the patch)

Discussion thread closes the loop: in current master (b9305+), `--n-cpu-moe` on llama-bench / llama-server reproduces the same effect without rebuilding. The patch is no longer required for users; it is preserved as the explanatory artifact.

```bash
GGML_CUDA_ENABLE_UNIFIED_MEMORY=1 \
./build/bin/llama-server \
  -m DeepSeek-V3.1-Q4_K_M.gguf \
  -ngl 99 \
  --n-cpu-moe 58 \
  -c 32768 \
  -b 4096 -ub 4096 \
  -fa on \
  --jinja --no-mmap \
  -t 32 \
  --host 0.0.0.0 --port 8080
```

Notes:
- DeepSeek V3.1 has 61 MoE layers (3 dense FFN at start, 58 MoE). `--n-cpu-moe 58` puts **all 58 MoE expert tensors** in Grace LPDDR; the 3 dense + attention + shared-expert + KV stays in HBM.
- `-t 32` is the established sweet spot from #18005 (Grace has 72 cores, but 32 maximizes MoE thread fanout before saturating the memory subsystem).
- `-b 4096 -ub 4096` is the Doctor-Shotgun guide recommendation for MoE; defaults `-b 2048 -ub 512` are too small for CPU+GPU split.
- `--no-mmap` important on aarch64 with UM — mmap interferes with cudaMemAdvise hints.

### 1.7 Batched / concurrent numbers (the 499 t/s figure)

The same thread reports `llama-batched-bench` at batch size 32, 8192-token prompts:

```bash
GGML_CUDA_ENABLE_UNIFIED_MEMORY=1 \
./bin/llama-batched-bench \
  -m DeepSeek-V3.1-Q4_K_M.gguf \
  -fa 1 -c 300000 -ub 2048 \
  -npp 512,4096,8192 -ntg 32
```

Result: at **batch=32, prompt 8192, gen 32**, aggregate throughput is **499.43 t/s** (PP+TG combined across 32 concurrent sequences). Translated: each user sees ~15.6 t/s when 32 users are batched. That is **27× the single-stream TG** in terms of aggregate work — MoE batching scales well on GH200 because the expert read amortizes across concurrent tokens that route to the same expert. The bottleneck flips from C2C-bound to compute-bound as batch grows.

## 2. Can vLLM 0.21 or SGLang 0.5.12 serve DeepSeek V3.1 671B on a single GH200?

**Short answer: not well in May 2026.** vLLM and SGLang are TP-first MoE runtimes; their CPU-offload paths for expert weights are documented as broken/partial; and neither has published a single-GH200 number for 671B-class MoE.

### 2.1 vLLM 0.21 single-GH200 — the failure modes

- **`--cpu-offload-gb` does not work for DeepSeek-V*-class MoE.** Issue **#15196** ("Does vLLM support MoE model use --cpu-offload-gb") was **closed as not planned, stale 90+ days**. Issue **#15871** ("CPU offload not working for DeepSeek-V2-Lite-Chat") shows the device-mismatch crash (`Expected all tensors to be on the same device, but found at least two devices, cuda:0 and cpu!`). Issue **#32653** (Unsloth DeepSeek R1 IQ1_S 1.58B GGUF) shows OOM even with `--cpu-offload-gb=100` on 4×A5000; the offload allocator fails inside `DeepseekV3ForCausalLM` MoE layer init.
- **RFC #33869 "DeepSeek-R1 MoE offload"** (opened 2026-02-05, **status: open, not merged as of 2026-05-25**) proposes the *actual* mechanism: pinned-memory weight streaming with DBO (Dual Batch Overlap) or Prefetch mode, AVX/AMX-optimized CPU kernels. No benchmark numbers in the RFC. No GH200 mention. This is the path forward but **does not exist in shipped vLLM 0.21 today**.
- **`--enable-expert-parallel` requires TP_SIZE × DP_SIZE ≥ 2 GPUs.** Per vLLM docs: "EP_SIZE = TP_SIZE × DP_SIZE." Single-GH200 → EP=1 → defeats the purpose. The vLLM-recommended single-node deployment for DeepSeek V3 is `--tensor-parallel-size 1 --data-parallel-size 8 --enable-expert-parallel` — **8 GPUs minimum**.
- **Lambda's published GH200 + vLLM DeepSeek recipe uses 16× GH200**, not 1. Quote: "We deployed vLLM 0.7.2 with `--pipeline-parallel-size=4 --tensor-parallel-size=4`" → 16 GH200s in pipeline+tensor parallel, reaching **400 tok/sec aggregate** with **10 tok/sec/query** at `--max-num-seqs=64`. **No single-GH200 path published.**

vLLM does have **aarch64 wheels** as of v0.21 (May 15, 2026). The MoE expert-offload path is the blocker, not the build.

### 2.2 SGLang 0.5.12 single-GH200 — same blocker, different doors

- DeepSeek V3/V3.1/R1 is officially supported (`docs.sglang.io/basic_usage/deepseek_v3.html`) but the recommended deployment is **TP=8 on a single H100/H200 node** or multi-node. No `--cpu-offload-gb`-equivalent flag for MoE experts in 0.5.12.
- DeepSeek V4 support exists (v0.5.12) but #23743 targets **GB200** (Blackwell), not GH200. TileLang ARM64 fixes share lineage; whether they unlock single-GH200 671B is **unverified**.
- The SGLang `KTransformers` integration ticket (#11425, filed Oct 2025) **was closed inactive**. The standalone LMSYS blog (`lmsys.org/blog/2025-10-22-KTransformers/`) shows KTransformers + SGLang reaching 227.85 t/s on 8× L20 + Intel Xeon AMX — but **AMX, not Neoverse SVE**. KTransformers' CPU kernels are AMX-specific (Sapphire Rapids+). They will not light up on Grace's V2 cores. **Effective dead end for GH200**.

### 2.3 Net conclusion (single GH200 + 671B MoE)

| Engine | Can serve DeepSeek V3.1 on 1×GH200? | TG t/s | Notes |
|---|---|---:|---|
| **llama.cpp + `--n-cpu-moe`** | **Yes (verified)** | **18.3** | Headline. Headroom for concurrency to 499 t/s aggregate at batch=32. |
| vLLM 0.21 | No (MoE+CPU-offload broken, EP needs ≥2 GPUs) | — | RFC #33869 may fix in future release. |
| SGLang 0.5.12 | No (no expert CPU offload) | — | KTransformers integration closed, AMX-only anyway. |
| KTransformers standalone | Possible in principle, but CPU kernels are AMX | — | Will not vectorize well on Neoverse-V2. |

**llama.cpp is the only working path for single-GH200 + DeepSeek V3.1 671B in May 2026.**

## 3. Kimi K2 Thinking ~1T — exact recipe for 21.6 t/s

### 3.1 Source

Same #18005 thread, same GH200 144 GB HBM3e + Grace 480 GB LPDDR5X + CUDA 13.0.

### 3.2 Model file

- **Kimi-K2-Thinking Q3_K_M GGUF**
- File size: **456.34 GiB**, parameter count 1026.41 B
- The native model is INT4 QAT (Moonshot ships native INT4 weights) — Q3_K_M GGUF is a community re-quantization from the bf16 dequant
- Architecture: DeepSeek-derived MoE — 384 routed experts (Kimi K2 uses 8 routed + 1 shared per layer per token, ~32B active params)
- For "lossless" inference Unsloth recommends UD-Q8_K_XL but file size is ~1 TB; UD-Q4_K_XL is 384 GB; for fitting in GH200 + Grace combined, **Q3_K_M (456 GB) is the sweet spot** — fits with 24 GB headroom for KV/context

### 3.3 The command

```bash
GGML_CUDA_ENABLE_UNIFIED_MEMORY=1 \
./build/bin/llama-bench \
  -m Kimi-K2-Thinking-Q3_K_M.gguf \
  -fa 1 -p 2048 -n 32 -ub 2048
```

Result:
| Metric | Value |
|---|---:|
| pp2048 | **521.11 ± 1.41 t/s** |
| tg32 | **21.57 ± 0.04 t/s** |

**Kimi K2 TG (21.6 t/s) is faster than DeepSeek V3.1 TG (18.3 t/s)** despite Kimi being ~1.5× larger. Reason: K2's active-parameter routing is leaner (32B vs 37B active) **and** the model has fewer dense layers in the active path, so less data crosses C2C per token.

### 3.4 Production server-mode

```bash
GGML_CUDA_ENABLE_UNIFIED_MEMORY=1 \
./build/bin/llama-server \
  -m Kimi-K2-Thinking-Q3_K_M.gguf \
  -ngl 99 \
  --n-cpu-moe 60 \
  -c 16384 \
  -b 4096 -ub 4096 \
  -fa on --jinja --no-mmap \
  -t 32 \
  --host 0.0.0.0 --port 8080
```

K2 Thinking has ~61 layers with MoE at every layer except possibly first one; `--n-cpu-moe 60` is the safe default. Context: 256K native, but at 256K the KV cache will not fit in HBM — start at 16K, grow as VRAM allows.

### 3.5 What the alternative engines do with Kimi K2

- vLLM's official Kimi-K2 recipe (`docs.vllm.ai/projects/recipes/en/latest/moonshotai/Kimi-K2.html`) requires **16× H800 minimum** with `--tensor-parallel-size 8 --pipeline-parallel-size 2 --quantization fp8`. **Single GH200: not supported.**
- Simplismart's 173 t/s number is on **8× B200 NVFP4** (Blackwell), not GH200.
- SGLang same constraints (16× H800 floor for Kimi K2).

**llama.cpp on a single GH200 is the cheapest way in 2026 to run Kimi K2 Thinking at any throughput at all.** 21.6 t/s is below what 16×H800 gives you, but it's on **one GPU** for ~$30K vs $250K+.

## 4. GPT-OSS 120B MXFP4 — 5,393 t/s PP, 209 t/s TG

### 4.1 Architecture

- **120B total, 5.1B active** per token (top-4 of 128 experts per layer)
- 36 layers, 2880 residual width, GQA + RoPE, **alternating dense + sparse attention**
- Trained natively in MXFP4 (~Q4_0 equivalent in GGUF terms) — quantization is native, not post-hoc

### 4.2 Model file

- GPT-OSS 120B MXFP4 GGUF, ~63 GB on disk (fits **entirely in HBM3e** — no offload needed)
- This is the critical difference from DeepSeek/Kimi: **GPT-OSS 120B has so few active params that the full Q4-class weights fit in the 144 GB HBM**

### 4.3 The command

```bash
./build/bin/llama-bench \
  -m gpt-oss-120b-MXFP4.gguf \
  -fa 1 -p 2048 -n 32 -ub 2048
```

Result (CUDA-only, no UM, no `--n-cpu-moe`):
| Metric | Value |
|---|---:|
| pp2048 | **5393.05 ± 24.60 t/s** |
| tg32 | **209.01 ± 1.06 t/s** |

**No unified memory tricks needed.** This is a vanilla `-ngl 99` HBM-resident run. The numbers are 10–25× higher than DeepSeek V3.1 because zero C2C traffic per token.

### 4.4 Production server-mode

```bash
./build/bin/llama-server \
  -m gpt-oss-120b-MXFP4.gguf \
  -ngl 99 \
  -c 131072 \
  -b 4096 -ub 4096 \
  -fa on --jinja \
  -t 16 --parallel 16 \
  --host 0.0.0.0 --port 8080
```

128K context — fits cleanly in the remaining ~80 GB HBM after weights. `--parallel 16` should be safe; per-user TG will degrade roughly linearly until compute-bound around 32–64 concurrent.

### 4.5 The 20B sibling for context

| Mode | GPT-OSS 20B pp2048 | GPT-OSS 20B tg32 |
|---|---:|---:|
| CPU 32 thr | 216.27 | 84.52 |
| CPU 72 thr | 377.55 | 34.83 (regression) |
| CUDA -ngl 99 | **9682.19** | **322.93** |

Two takeaways:
1. The 72-thread regression (84.52 → 34.83) is the same memory-subsystem-saturation artifact that drove the 32-thread choice for V3.1/K2.
2. The 20B → 120B comparison (322.93 → 209.01 t/s TG) is a ~35% slowdown for 6× the parameters — exactly what MoE sparse routing should give you. Active set growth (~1.5B → 5.1B = 3.4×) is the dominant factor, not total params (6×).

## 5. Mixtral 8x22B (141B / 39B active) and Qwen3-Next-80B-A3B-Thinking

### 5.1 Mixtral 8x22B on GH200

- **141B total, 39B active** (2 of 8 experts top-k routing; ~25% activation density per layer)
- bf16 → 282 GB. **Does not fit in 144 GB HBM** but fits with C2C offload of experts.
- Q4_K_M GGUF → ~80 GB; **fits entirely in HBM** on 144 GB SKU, snug on 96 GB SKU.
- No published GH200-specific tok/s number in May 2026. Extrapolating from #18005:
  - Q4_K_M, all-HBM, single stream: expect ~50–70 t/s TG (between GPT-OSS 120B's 209 t/s and a dense 70B's 35 t/s; 39B active is closer to 40B dense, slightly faster because MoE)
  - bf16 with `--n-cpu-moe`: probably 25–35 t/s TG (active set is much fatter than DeepSeek's, so more C2C per token)
- **vLLM on single GH200**: Q4_K_M is not supported natively; bf16 needs ~282 GB — does not fit in HBM, would need expert offload (broken). FP8 quant (162 GB) still doesn't fit HBM, and FP8 is flagged broken on this stack anyway. **vLLM Mixtral 8x22B on single GH200 in bf16/FP8: blocked**.
- **Engine ranking for Mixtral 8x22B on single GH200**: llama.cpp Q4_K_M (works, ~60 t/s estimated) >> vLLM bf16 expert offload (blocked) >> SGLang (same blockers).

### 5.2 Qwen3-Next-80B-A3B-Thinking

- **80B total, only 3B active** (hybrid MoE — extremely sparse; one of the lightest active sets in the 100B class)
- bf16 → 160 GB. **Fits in 144 GB HBM only with quant**; Q4_K_M → ~45 GB → fits easily on either 96 or 144 GB SKU.
- **vLLM recipe** (`docs.vllm.ai/projects/recipes/en/latest/Qwen/Qwen3-Next.html`) explicitly states "4× H200/H20 or 4× A100/A800 GPUs to launch this model" with `--tensor-parallel-size 4`. **Single GH200 is not in the official recipe**, but the 4×H200 recommendation is for bf16; at Q4 a single GH200 has the memory and may work via the Qwen3-Next MTP speculative path (`--speculative-config '{"method": "qwen3_next_mtp", "num_speculative_tokens": 2}'`).
- **llama.cpp**: Q4_K_M GGUF available, all-HBM, single stream: expect 70–120 t/s TG. The 3B active set is even more cache-friendly than GPT-OSS's 5.1B. **No published GH200 number yet**; this is a high-value missing benchmark.
- Known issue: `ggml-org/llama.cpp#19480` ("Qwen3-Coder-Next CPU inference ~5x slower than expected") — Qwen3-Next's hybrid linear-attention layout has a CPU-path perf bug. Should not affect CUDA-on-GH200, but worth watching.

## 6. Concurrent serving — can a single GH200 serve >1 user?

### 6.1 llama.cpp answer (positive)

llama.cpp has had continuous batching since 2024 (`--cont-batching` default-on, `--parallel N` / `-np N` slots). Discussion #18308 ("Optimal parameters for parallel inference using llama-server") and the Doctor-Shotgun MoE guide confirm:
- Per-user KV at 8K context for MoE with GQA: **~200 MB/slot** (Qwen3.5-35B-A3B measurement; DeepSeek V3.1 will be larger due to MLA — call it ~400 MB/slot)
- Continuous batching merges requests into the `-b 4096`/`-ub 4096` budget; effective per-user TG falls roughly linearly until you saturate compute or memory bandwidth

#### Measured #18005 numbers (batched-bench)

For **DeepSeek V3.1 Q4_K_M, batch=32, 8192 prompt + 32 gen**:
- **Aggregate throughput: 499.43 t/s** (vs 18.30 t/s single-stream)
- **Per-user effective: ~15.6 t/s** at 32 concurrency
- **Implied amortization**: 32 users get 27.3× the work of 1 user → 85% scaling efficiency at 32 batch

For **GPT-OSS 120B MXFP4, batch=32, 8192 prompt**:
- Implied aggregate from #18005 (extrapolated from PP 5393 t/s): probably ~1500–2500 t/s aggregate TG at batch=32 with 209 t/s single-stream baseline → 7–12× amortization (smaller because already compute-bound at single stream)

#### Practical recipe for multi-user GH200 serving

```bash
# DeepSeek V3.1, target 32-user MoE serving:
GGML_CUDA_ENABLE_UNIFIED_MEMORY=1 \
./llama-server \
  -m DeepSeek-V3.1-Q4_K_M.gguf \
  -ngl 99 --n-cpu-moe 58 \
  -c 65536 \                    # 2K per user × 32 parallel
  --parallel 32 \
  -b 4096 -ub 4096 \
  -fa on --jinja --no-mmap \
  -t 32 \
  --host 0.0.0.0 --port 8080

# GPT-OSS 120B, target 16-user low-latency:
./llama-server \
  -m gpt-oss-120b-MXFP4.gguf \
  -ngl 99 \
  -c 65536 \
  --parallel 16 \
  -b 4096 -ub 4096 \
  -fa on -t 16 \
  --host 0.0.0.0 --port 8080
```

### 6.2 vLLM/SGLang single-GH200 — no path because model doesn't fit at all

Until vLLM RFC #33869 ships, vLLM cannot serve 671B-class MoE on a single GH200 at any concurrency, batched or not. The MoE-CPU-offload bug applies independent of `--max-num-seqs`.

### 6.3 Memory budget for concurrency

GH200 144 GB SKU: after 96 GB HBM holds attention+dense+norms+KV-overhead for DeepSeek V3.1, ~50 GB remains for KV cache and active expert slots. At 400 MB/slot, **~120 concurrent users at 8K context** is the theoretical memory ceiling. **Compute will saturate long before** — probably at batch 32–64.

## 7. Active-parameter routing efficiency — how much of Grace LPDDR is actually touched per token?

### 7.1 The arithmetic

For DeepSeek V3.1 Q4_K_M (377 GB on disk, ~95% of which is in expert tensors):
- **Active per token**: 1 shared + 8 routed experts × 58 MoE layers = 522 expert reads per forward pass + 3 dense FFN + attention
- **Per-expert weight footprint at Q4**: each routed expert is ~610 MB (4-bit quantized, gate/up/down × 7168 hidden × 2048 intermediate)
- **Active expert weight per token**: 522 reads × 610 MB ÷ 9 experts/layer used per token = roughly **~12–15 GB of unique expert weight is "live" per token on average across the forward pass**
- (This is a rough estimate; the real number depends on tokenizer + routing-locality)

### 7.2 Effective bandwidth

If TG = 18.3 t/s and ~12–15 GB of LPDDR5X is read per token, then:
- **Effective C2C/LPDDR read bandwidth in the active path: ~220–275 GB/s**
- This is **40–50% of the 546 GB/s LPDDR5X nominal** and **49–61% of the 450 GB/s C2C nominal**
- The verda.com STREAM measurement of ~297 GB/s D2H and ~375 GB/s H2D bounds this — the active path is **bandwidth-bound on C2C reads**, not on Grace DRAM itself

### 7.3 Implications

- **Per-token, only ~3% of the model is "touched"** (12–15 GB out of 378 GB) — the rest sits cold in Grace LPDDR
- This is why MoE on GH200 is fundamentally well-suited: you carry the full weight set in cheap LPDDR5X (480 GB @ ~$1500-equivalent cost), pay HBM-equivalent bandwidth only for the ~3% active set per token
- **Concurrent batching helps massively** because tokens within a batch often route to overlapping experts → cache locality in HBM-side staging buffers (this is the mechanism behind the 27× aggregate scaling at batch=32)
- The cudaMemAdvise patch's whole reason for existing is to **prevent CUDA's UVM from migrating cold expert pages into HBM**, which would waste HBM and thrash the active path

### 7.4 Why this won't change with newer GH200 SKUs

- 144 GB HBM3e SKU has slightly more HBM bandwidth (4.9 TB/s vs 4.0 TB/s) but **C2C is unchanged at 900 GB/s bidirectional / 450 GB/s per direction**
- The bottleneck for 671B-class MoE on GH200 is C2C, not HBM. **Faster HBM doesn't help; faster C2C would** — Grace-Blackwell (GB200) has 1.8 TB/s C2C, which is the architectural answer
- For models that fit in HBM (≤141B Q4), the HBM bandwidth advantage matters; for 671B+ MoE, you're C2C-bound and would gain more from GB200 than from H200

## 8. Speculative decoding for MoE on GH200 — current state

- llama.cpp PR #22673 "llama + spec: MTP Support" (multi-token prediction draft heads) — landing as of late May 2026, **not in stable b9305 yet**.
- DeepSeek V3/V3.1 ships with **native MTP heads** in its architecture (one extra layer predicting tokens +1, +2, etc). The llama.cpp MTP PR enables these heads as the draft for speculative decode **without a separate draft model**.
- Expected speedup on dense models: 1.5–2× (community benchmarks). On MoE with `--n-cpu-moe`, the speedup may be **lower** because the bottleneck is C2C-bound expert reads, not compute; MTP only helps if you were compute-bound.
- **No GH200-specific MTP benchmark exists yet**. This is a Round-5 target.

## 9. Practical decision tree (May 2026)

```
Do you need to serve a 100B–1T MoE on a single GH200?
│
├── Model fits in HBM at Q4 (≤141B total)?
│   ├── Yes  → llama.cpp -ngl 99, no UM, all-HBM
│   │         GPT-OSS 120B → 209 t/s; Mixtral 8x22B → ~60 t/s est;
│   │         Qwen3-Next-80B → ~80–120 t/s est
│   │
│   └── No (DeepSeek V3.1, Kimi K2, anything ≥250 GB at Q4)
│       │
│       ├── llama.cpp + GGML_CUDA_ENABLE_UNIFIED_MEMORY=1 + --n-cpu-moe N
│       │   DeepSeek V3.1 → 18.3 t/s TG; Kimi K2 → 21.6 t/s TG
│       │   Concurrent at batch=32 → 499 t/s aggregate (DeepSeek)
│       │   THIS IS THE ONLY WORKING PATH IN MAY 2026
│       │
│       ├── vLLM 0.21 → BLOCKED. MoE+CPU-offload bug (#15871, #32653).
│       │   RFC #33869 (open, not merged) is the future path.
│       │
│       └── SGLang 0.5.12 → BLOCKED. KTransformers integration (#11425)
│                            closed inactive. AMX-only anyway.
│
└── Need >32 concurrent users?
    └── Stay with llama.cpp + --parallel 32-64; per-user TG drops to 5-15 t/s
        but aggregate scales near-linearly until compute-bound.
```

## 10. Confidence & gaps

### High confidence
- DeepSeek V3.1 Q4_K_M 18.30 t/s TG, Kimi K2 Q3_K_M 21.57 t/s TG, GPT-OSS 120B MXFP4 209.01 t/s TG on GH200 144 GB — directly from #18005 with reproducible command lines
- DeepSeek V3.1 batch=32 aggregate 499.43 t/s — from the same thread
- vLLM `--cpu-offload-gb` does not work for DeepSeek-family MoE (issues #15871, #32653, #15196 closed-not-planned)
- vLLM Kimi K2 official recipe needs ≥16 GPUs
- GPT-OSS 120B architecture: 36 layers, 128 experts, top-4, 5.1B active
- Active-parameter sparsity: ~3% of total weights touched per token in DeepSeek V3.1
- Effective C2C bandwidth utilization: 40–50% of nominal LPDDR5X, 50–60% of nominal C2C — bandwidth-bound on C2C reads, not Grace DRAM
- `--n-cpu-moe N` replaces the cudaMemAdvise patch in current master

### Medium confidence
- Mixtral 8x22B Q4_K_M on GH200 ~50–70 t/s and Qwen3-Next-80B Q4 ~80–120 t/s — extrapolated from active-parameter scaling, not measured
- Per-user effective TG at batch=32 (~15.6 t/s for DeepSeek V3.1) — from aggregate÷32 algebra, assumes uniform routing
- MTP speculative-decode speedup on GH200 MoE will be lower than dense because C2C-bound — reasoned, not measured
- 12–15 GB live expert weight per token for DeepSeek V3.1 — estimate based on 9 experts × 58 layers × 610 MB/expert / 9 reads, may be off by ±50%

### Low confidence / open gaps
- **No published vLLM-on-single-GH200 number for any 100B+ MoE model** — Lambda's published recipe is 16×GH200. Round 5 should attempt a manual single-GH200 vLLM bf16 fit of GPT-OSS 120B (it fits HBM, no MoE-offload bug needed) to anchor a vLLM number.
- **No published SGLang-on-single-GH200 number for any 100B+ MoE** — same gap as Round 3.
- **No GH200-on-Mixtral-8x22B published number** at all. Round 5 should run it.
- **Qwen3-Next-80B-A3B on GH200**: officially recommended ≥4 GPUs but Q4 fits in 144 GB HBM; needs an actual measurement.
- **MTP-on-GH200 speculative decode speedup**: no benchmark, no estimate worth committing to.
- **KV cache memory footprint at 128K context for DeepSeek V3.1 on GH200** is not directly cited in the source; MLA compression helps but the exact GB/slot is unverified.
