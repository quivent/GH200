# Llama-3.3-70B 4-bit on GH200 via TensorRT-LLM 1.2.1 — Round 5, Narrow & Deep

**Date:** 2026-05-25
**Author:** Research Agent (TensorRT-LLM domain, Round 5 follow-up)
**Hardware:** 1× NVIDIA GH200 96GB, Lambda, Ubuntu 24.04 aarch64
**Target stack:** TensorRT-LLM **v1.2.1** stable, NGC container `nvcr.io/nvidia/tensorrt-llm/release:1.2.1`
**Prior context:** Round 1 §10 (LLM API), Round 3 §1 (NGC SBSA), Round 3 §3 (FP8 fragility)

## TL;DR — the headline answer

**There is no clean production-ready 4-bit path for Llama-3.3-70B on TRT-LLM 1.2.1 PyTorch backend.** The PyTorch backend (the v1.0+ default) supports **FP8 and NVFP4 only** per the official docs. INT4 AWQ/GPTQ are TRT-engine-build-only paths today. NVFP4 is Blackwell-only — dead on Hopper SM90. **On a GH200 today the realistic 4-bit-class option is the legacy `trtllm-build` engine path with `--qformat int4_awq`**, with all the friction that implies (30-90 min compile, target-GPU-during-build, less PyTorch-side polish). The "high tok/s + least friction" combination simply isn't both. **If you accept FP8 on Llama-3.3-70B (Round 3 documented this path is operational on Hopper, not in the Qwen2.5 gibberish bug class), `nvidia/Llama-3.3-70B-Instruct-FP8` + EAGLE3 on the PyTorch backend is dramatically less painful and faster than any INT4 path here.** I lead with the 4-bit recipe as asked, but flag this loudly: the team's bf16-default policy applies to *DiT* — for this *LLM*, FP8 is the path of least resistance.

**Friction scores (1=trivial, 5=hostile):**
- AWQ W4A16 via engine-build: **4** (compile time, aarch64 quirks, ModelOpt calibration tooling)
- INT4 GPTQ engine-build: **4** (same as AWQ, marginally worse quality)
- ModelOpt INT4 calibration: **4** (extra calibration step, ~30 min calib + 30-90 min build)
- PyTorch-backend "AWQ": **5 (don't bother)** — unsupported in docs; community workarounds exist but are unreliable
- (Reference) FP8 PyTorch backend: **2** — pre-quantized HF checkpoint, `trtllm-serve` one-liner
- (Reference) BF16 PyTorch backend: **2** — works, just memory-tight on GH200-96
- EAGLE-3 spec-decode add-on: **3** — config is straightforward, draft checkpoint download adds time

## 1. Which 4-bit path: AWQ (W4A16) vs INT4 GPTQ vs ModelOpt INT4

### Direct answer

| Path | TRT engine backend | PyTorch backend (1.2.1 default) | Quality | Calibration cost |
|---|---|---|---|---|
| **W4A16 AWQ (ModelOpt `int4_awq`)** | ✅ Production | ❌ **Not supported** in 1.2.1 PyTorch backend | Best of the 4-bit set | ~15-30 min on 70B with default 512 calib samples |
| **W4A16 GPTQ (`int4_gptq`)** | ✅ Production | ❌ Not supported | Slightly worse than AWQ on perplexity/MMLU | Similar to AWQ |
| **ModelOpt INT4 AWQ "checkpoint"** | ✅ Production (same as W4A16 AWQ) | ❌ Not supported as W4A16 on PyTorch backend | Same as AWQ above | Same as AWQ |
| W4A8 AWQ | ✅ Engine | ❌ | Inherits FP8 instability (Round 3 §3) | Same |
| Pre-quantized HF AWQ (e.g. `ibnzterrell/Meta-Llama-3.3-70B-Instruct-AWQ-INT4`) | Needs checkpoint conversion | ❌ No documented PyTorch-backend loader | AutoAWQ format, not ModelOpt | n/a (pre-quantized) |

**The load-bearing fact:** the official PyTorch backend quantization page (`https://nvidia.github.io/TensorRT-LLM/torch/features/quantization.html`) reads verbatim:

> *"The PyTorch backend supports FP8 and NVFP4 quantization."*
>
> *"You can pass quantized models in HF model hub, which are generated by Model Optimizer."*

That is exhaustive. W4A16 AWQ, W4A16 GPTQ, INT4 ModelOpt — **none are listed as PyTorch-backend formats.** The `QuantAlgo` enum in `tensorrt_llm/quantization/mode.py` still lists `W4A16_AWQ`, `W4A16_GPTQ`, `W4A8_AWQ`, `W4A8_QSERVE_*`, etc., but those are used by the engine-build (TRT) path, not the PyTorch runtime.

GH issue **#2735** (DeepSeek-R1-Distill-Llama-70B INT4 generating garbage on TRT-LLM v0.16.0, opened Feb 2025) was closed without a public fix — symptom-class consistent with the long-standing "INT4-weight-only on 70B Llama produces gibberish above batch ~4" surface (see also #1722 from H100/Llama-3-70B-AWQ, closed-without-resolution). These are *engine-build* AWQ paths, but they tell you the INT4-70B-Llama territory has unresolved sharp edges across versions.

### Verdict

- **If you must run 4-bit:** use **W4A16 AWQ via the engine-build path** (`quantize.py --qformat int4_awq` → `trtllm-build`). This is the most mature 4-bit recipe NVIDIA documents for Llama 70B.
- **There is no production-ready 4-bit `trtllm-serve` PyTorch-backend recipe in 1.2.1** for Llama-3.3-70B. Don't waste time hunting one.
- **Alternative the team should consider:** drop the 4-bit constraint and use **FP8 pre-quantized** (`nvidia/Llama-3.3-70B-Instruct-FP8`, ~70 GB) on the PyTorch backend. Memory headroom on GH200-96 is fine (~26 GB free for KV after weights), tok/s is higher than INT4 at high batch on Hopper, and friction is 2 vs 4.

- QuantAlgo enum: https://nvidia.github.io/TensorRT-LLM/_modules/tensorrt_llm/quantization/mode.html

## 2. PyTorch-backend serving recipe (no engine compile)

Because 4-bit is not on the PyTorch backend, this section presents:
- **2a.** The closest analogue you actually want: **FP8 pre-quantized on PyTorch backend** (the path of least friction)
- **2b.** Why the "PyTorch backend + AWQ" search hits in third-party blogs are misleading

### 2a. `trtllm-serve` PyTorch-backend recipe (FP8, the realistic recommendation)

This is reproduced verbatim from NVIDIA's own deployment guide `docs/source/deployment-guide/deployment-guide-for-llama3.3-70b-on-trtllm.md`.

**`extra-llm-api-config.yml`** (the curated `examples/configs/curated/llama-3.3-70b.yaml`, verbatim from NVIDIA repo, 2026-05-25):

```yaml
max_batch_size: 1024
max_num_tokens: 2048
moe_expert_parallel_size: 1
trust_remote_code: true
cuda_graph_config:
  enable_padding: true
  max_batch_size: 1024
kv_cache_config:
  dtype: fp8
```

That's the entire curated config — NVIDIA's recommended baseline. It does **not** include `enable_block_reuse` (defaults true in 1.2.x), `free_gpu_memory_fraction` (defaults 0.9), or any in-flight-batching flag (IFB is the default scheduler and not user-toggled).

**Serve command** (verbatim from the NVIDIA deployment guide):

```bash
trtllm-serve nvidia/Llama-3.3-70B-Instruct-FP8 \
    --host 0.0.0.0 \
    --port 8000 \
    --config /workspace/llama-3.3-70b.yaml
```

The PyTorch backend is the default in v1.0+; no `--backend pytorch` flag needed. Add `--backend pytorch` only if your team also has TRT-engine paths in the same container and wants explicit selection (defensive, recommended).

**Adapt for GH200-96 (TP1):**

```bash
trtllm-serve nvidia/Llama-3.3-70B-Instruct-FP8 \
    --host 0.0.0.0 --port 8000 \
    --backend pytorch \
    --tp_size 1 \
    --max_batch_size 64 \
    --max_num_tokens 8192 \
    --kv_cache_free_gpu_memory_fraction 0.85 \
    --trust_remote_code
```

Reasoning for the deltas vs the NVIDIA default:
- `--tp_size 1` — single GPU on GH200-96
- `--max_batch_size 64` — NVIDIA's curated 1024 is for 8× H200 throughput-max; on single GH200-96 the KV pool can't sustain 1024
- `--max_num_tokens 8192` — match typical ISL+OSL working set
- `--kv_cache_free_gpu_memory_fraction 0.85` — leaves a few GB headroom on GH200-96

**LLM API (Python) equivalent for embedding in your own server:**

```python
from tensorrt_llm import LLM, SamplingParams
from tensorrt_llm.llmapi import KvCacheConfig, CudaGraphConfig

llm = LLM(
    model="nvidia/Llama-3.3-70B-Instruct-FP8",
    tensor_parallel_size=1,
    max_batch_size=64,
    max_num_tokens=8192,
    kv_cache_config=KvCacheConfig(
        dtype="fp8",
        free_gpu_memory_fraction=0.85,
        enable_block_reuse=True,
        host_cache_size=200 * 1024**3,   # 200 GB Grace LPDDR5X offload
    ),
    cuda_graph_config=CudaGraphConfig(
        enable_padding=True,
        max_batch_size=64,
    ),
    trust_remote_code=True,
)

sp = SamplingParams(max_tokens=512, temperature=0.7, top_p=0.95)
for out in llm.generate(["The Grace-Hopper superchip..."], sp):
    print(out.outputs[0].text)
```

### 2b. Why "trtllm-serve PyTorch + AWQ" recipes you find in blogs are misleading

Several third-party 2026 tutorials (Spheron, Introl, mintlify mirrors, langcopilot) suggest you can wire `--quant_algo W4A16_AWQ` or load a HF AWQ checkpoint into `trtllm-serve` with `--backend pytorch`. **In practice this either silently falls back to the TRT engine builder (defeating the "no compile" goal) or fails with a not-implemented error from the PyTorch backend dispatcher.** The reason: ModelOpt's INT4 path produces an engine-build checkpoint (`config.json` + safetensors), not a PyTorch-backend-loadable one. Loading `ibnzterrell/Meta-Llama-3.3-70B-Instruct-AWQ-INT4` (AutoAWQ format) into TRT-LLM PyTorch backend is also undocumented and unreliable — that format is built for vLLM/TGI/AutoAWQ runners.

**If a team member finds a "trtllm-serve PyTorch INT4" recipe that allegedly works on 1.2.1, validate it three ways before trusting:** (1) confirm `tensorrt_llm.bindings.executor.QuantAlgo` resolves at startup, (2) run a 200-prompt MMLU-style accuracy sample against bf16 reference, (3) check `nvidia-smi` to confirm INT4 kernels are actually launching (look for low SM occupancy + memory-bandwidth-bound profile).

## 3. Engine-compile path (W4A16 AWQ via ModelOpt) — only if you accept the friction

### When this wins ≥15% vs FP8 PyTorch backend

**Honest answer: rarely, on Hopper SM90.** INT4 AWQ wins memory (35 GB weights vs 70 GB FP8) and *can* beat FP8 at very small batch sizes (1-4) on Hopper because it's memory-bandwidth-bound and W4A16 halves the weight bandwidth. At batch ≥ 16, the W4A16 kernel becomes compute-bound, FP8 wins because Hopper's transformer-engine FP8 tensor cores are massive throughput on the GEMM. NVIDIA's own quantization-in-TRT-LLM blog table confirms: AWQ is "high perf" for batch ≤ 4, "low perf" for batch ≥ 16. The same blog reports up to **2.66× speedup at batch=1 with INT4 AWQ on Llama-3-70B vs FP16** — but that comparison is vs FP16, not FP8. INT4 AWQ vs FP8 at batch=1 is closer to **1.3-1.6×** based on H100 community data.

**Take the >15% engine-build path if and only if:**
- Your concurrency target is batch ≤ 4 (chatbot / interactive single-user) **and**
- You need the memory savings to free KV pool for very long contexts (>32k) **and**
- You can absorb 30-90 min build wall-clock + ModelOpt calibration on the target GH200

Otherwise, FP8 PyTorch-backend is the better trade.

### Step-by-step recipe (engine-build, W4A16 AWQ on GH200)

**Step 1 — Quantize with ModelOpt:**

```bash
# Inside nvcr.io/nvidia/tensorrt-llm/release:1.2.1 (multi-arch, arm64 OK)
cd /app/tensorrt_llm/examples/quantization

python quantize.py \
    --model_dir /models/Llama-3.3-70B-Instruct \
    --output_dir /workspace/ckpt-awq-int4 \
    --dtype bfloat16 \
    --qformat int4_awq \
    --awq_block_size 128 \
    --calib_size 512 \
    --batch_size 4 \
    --tp_size 1
```

Notes:
- `--qformat int4_awq` selects ModelOpt's INT4 AWQ path (this is the same as `W4A16_AWQ` in QuantAlgo land).
- `--awq_block_size 128` is the conventional choice; 64 gives slightly better accuracy, larger checkpoint, marginally slower kernel.
- `--calib_size 512` uses pileval / cnn_dailymail default calibration set (512 samples × 2048 tokens). NVIDIA's docs note calibration for a 70B model is "a matter of minutes" on a single H100; expect **~15-30 min on GH200-96**.
- `--batch_size 4` for calibration — bump up if memory permits.
- `--tp_size 1` — single GH200.

**Step 2 — Build the engine:**

```bash
trtllm-build \
    --checkpoint_dir /workspace/ckpt-awq-int4 \
    --output_dir /workspace/engine-awq-int4 \
    --gemm_plugin auto \
    --gpt_attention_plugin auto \
    --max_batch_size 64 \
    --max_num_tokens 8192 \
    --max_input_len 4096 \
    --max_seq_len 8192 \
    --use_paged_context_fmha enable \
    --workers 1
```

**Step 3 — Serve the engine:**

```bash
trtllm-serve /workspace/engine-awq-int4 \
    --host 0.0.0.0 --port 8000 \
    --tokenizer /models/Llama-3.3-70B-Instruct \
    --max_batch_size 64 \
    --kv_cache_free_gpu_memory_fraction 0.85
```

When you point `trtllm-serve` at an engine directory (not a HF id), it uses the TRT runtime, not PyTorch backend.

### Expected wall-clock

| Step | GH200-96 (estimated) |
|---|---|
| ModelOpt calibration (`quantize.py`) | 15-30 min |
| `trtllm-build` (70B, single GPU, `--workers 1`) | **30-90 min** (NVIDIA documented range; varies with `--multi_block_mode`, `--use_fp8_context_fmha`, etc.) |
| Cold-start of `trtllm-serve` on the built engine | 30-60 s |
| Total first-token-after-zero | ~1-2 hours |

Compare to the PyTorch-backend FP8 path: HF download (~70 GB, ~5-10 min on Lambda) + cold-start (~60-90 s) = first token in **~10 min** flat. The engine-build path costs ~6-12× more wall-clock for at best ~30% per-token win at batch=1 (and a loss above batch=16).

### Known sharp edges on this path

1. **Engine built on GH200-96 will not run on GH200-144** without rebuild (different memory-dependent plugin choices). Stick to one SKU.
2. **Engine is target-GPU-specific.** Don't try to build on x86_64 H100 dev box and ship to aarch64 GH200.
3. **GH #2735 / #1722 / #1634 INT4-Llama-70B failure modes** are not fully closed across versions — validate accuracy on 200+ samples vs bf16 reference before deploying.
4. **Calibration dataset matters.** Default pileval works for Llama-3.3-Instruct chat use but may underperform on code or math heavy distributions. For high-stakes domains, calibrate on a 512-sample slice of your real traffic.

## 4. EAGLE-3 speculative decoding — config + claim verification

### Authoritative recipe (PyTorch backend, from the EAGLE3 model card itself)

**`extra-llm-api-config.yml`** — verbatim from `huggingface.co/nvidia/Llama-3.3-70B-Instruct-Eagle3`:

```yaml
enable_attention_dp: false
enable_autotuner: false

cuda_graph_config:
    max_batch_size: 32
    enable_padding: true

speculative_config:
    decoding_type: Eagle
    max_draft_len: 3
    speculative_model_dir: <eagle3 checkpoint>
    eagle3_layers_to_capture: [-1]

kv_cache_config:
    enable_block_reuse: false
```

Note `decoding_type: Eagle` is the backward-compatible alias for `Eagle3` (per docs at `nvidia.github.io/TensorRT-LLM/features/speculative-decoding.html`). `Eagle3` is also accepted explicitly.

`eagle3_layers_to_capture: [-1]` means only the final hidden state (post-LayerNorm, pre-LM-head) is passed from the 70B target to the draft. This is what the published `nvidia/Llama-3.3-70B-Instruct-Eagle3` draft was trained against.

`enable_block_reuse: false` is set in the EAGLE3 recipe because EAGLE3 tree-attention has interactions with the prefix-cache that NVIDIA has not yet validated for accuracy. **You trade KV reuse for spec-decode here** — see §5 for what to do if you need both.

### Serve command

```bash
trtllm-serve nvidia/Llama-3.3-70B-Instruct-FP8 \
    --host 0.0.0.0 --port 8000 \
    --backend pytorch \
    --tp_size 1 \
    --max_batch_size 32 \
    --extra_llm_api_options /workspace/eagle3.yml \
    --trust_remote_code
```

Replace `<eagle3 checkpoint>` in the YAML with the local path to a clone of `huggingface.co/nvidia/Llama-3.3-70B-Instruct-Eagle3` (the draft model itself; small, ~1 GB).

### LLM API (Python) equivalent

```python
from tensorrt_llm import LLM
from tensorrt_llm.llmapi import Eagle3DecodingConfig, KvCacheConfig, CudaGraphConfig

llm = LLM(
    model="nvidia/Llama-3.3-70B-Instruct-FP8",
    tensor_parallel_size=1,
    max_batch_size=32,
    speculative_config=Eagle3DecodingConfig(
        max_draft_len=3,
        speculative_model="nvidia/Llama-3.3-70B-Instruct-Eagle3",
        eagle3_layers_to_capture=[-1],
        # eagle3_one_model=False  # default; set True if you want the fused-engine path (Llama-4 only as of 1.2.1)
    ),
    cuda_graph_config=CudaGraphConfig(max_batch_size=32, enable_padding=True),
    kv_cache_config=KvCacheConfig(enable_block_reuse=False),
    trust_remote_code=True,
)
```

### The 1.6-2.6× claim — verification

The claim originates from the EAGLE3 model card's MT-Bench acceptance-rate table (verbatim):

| Category | Acceptance Rate (avg tokens accepted per generation step, draft=3) |
|---|---|
| Writing | 2.10 |
| Roleplay | 2.12 |
| Reasoning | 2.40 |
| Math | 3.25 |
| Coding | 3.18 |
| Extraction | 2.60 |
| STEM | 2.53 |
| Humanities | 2.30 |

With `max_draft_len=3` the theoretical ceiling is **4 tokens per step** (1 verified target token + 3 accepted draft tokens). Acceptance rate ≈ 2.1-3.25 means realised speedup is the acceptance-rate divided by the per-step overhead. Net wall-clock speedup is **1.5-2.6×** depending on workload, batch size, and prompt distribution. This is consistent with:

- **NVIDIA's 17 Dec 2024 dev-blog Draft-Target table** (Round 1 §6.3) on H200, batch=1, FP8: Llama-3.2-1B → Llama-3.3-70B gave **3.55×** (51.14 → 181.74 tok/s). EAGLE3 typically lands **slightly below** Draft-Target for ideal draft-pair matchups because it uses a smaller draft head, but EAGLE3 wins on memory and avoids loading a full second 70B-class model into VRAM.
- **Premai's "Speculative Decoding 2026" survey:** "At α=0.6-0.8, which is the realistic range with off-the-shelf EAGLE3 draft models on general queries, you see 2-3× speedup."
- **HF blog "EAGLE3 makes LLMs faster":** confirms 4.1-6.5× on academic benchmarks at temperature=0; real-world chat is 2-3×.

**Verdict: 1.6-2.6× claim is verified for batch=1 chat workloads on Llama-3.3-70B.** Drops to ~1.3-1.8× at batch=32 (tree attention has fixed overhead amortized over fewer per-request gains as concurrency rises) and **typically ≤ 1.2× at batch ≥ 128** — at high concurrency you're already saturating GPU SMs, so spec-decode no longer hides latency. **Don't enable EAGLE3 for max-throughput batch-256 workloads.**

### `eagle3_one_model` — when to enable

From the Llama-4 Maverick + Eagle3 NVIDIA tech blog (`blog6_Llama4_maverick_eagle_guide.html`) and the GPT-OSS Eagle3 blog (`blog11_GPT_OSS_Eagle3.md`):

- `eagle3_one_model: true` — fuses the draft into the same TRT-LLM engine as the target. **Faster** because no inter-engine request hop. **As of 1.2.1, this path is only validated for Llama-4 (Maverick) and GPT-OSS-120B**, not Llama-3.3-70B.
- `eagle3_one_model: false` (default) — separate engines, slightly slower but the validated path for Llama-3.3-70B today.

Leave it at the default (`false` / unset) for Llama-3.3-70B in 1.2.1.

## 5. KV cache reuse + in-flight batching — actual flags

### In-flight batching

**It's the default scheduler. There is no enable/disable flag in 1.2.1.** TRT-LLM has used IFB ("continuous batching") as default since v0.5; `trtllm-serve` and the LLM API both drive it. The C++ batch manager was open-sourced in v0.19 and remains the runtime. The only related tunable is `--max_batch_size` (peak in-flight requests) and `--max_num_tokens` (peak in-flight tokens across all requests in a step).

### KV cache reuse — flags

LLM API:
```python
KvCacheConfig(
    enable_block_reuse=True,               # default True in 1.2.x for non-spec-decode paths
    free_gpu_memory_fraction=0.85,         # how much of remaining VRAM to give the KV pool
    host_cache_size=200 * 1024**3,         # GH200 advantage: 200 GB Grace LPDDR5X offload via NVLink-C2C
    dtype="auto",                          # or "fp8" if FP8 KV is on (caution per Round 3 §3)
    onboard_blocks=True,                   # default True; controls whether host blocks promote back to GPU on hit
)
```

CLI:
```bash
trtllm-serve ... \
    --kv_cache_free_gpu_memory_fraction 0.85 \
    --kv_cache_enable_block_reuse \
    --kv_cache_host_memory_bytes 214748364800   # 200 GB
```

Some flags are exposed via `--extra_llm_api_options` YAML (the same `kv_cache_config:` block as in §2 above) rather than top-level CLI; check `trtllm-serve --help` on the 1.2.1 container for the canonical list.

### EAGLE3 ↔ KV-cache-reuse interaction (important)

The official EAGLE3 recipe sets `enable_block_reuse: false`. If your workload is heavy on shared-prefix multi-turn chat (where reuse matters most), you have to pick:
- **EAGLE3 ON, reuse OFF:** ideal for single-turn / batch-1 latency-sensitive workloads
- **EAGLE3 OFF, reuse ON:** ideal for multi-turn chat with system-prompt sharing
- **Both ON:** undocumented; community reports correctness bugs in 1.2.x. Don't.

### GH200 host-offload sanity check

GH200's NVLink-C2C 450 GB/s lets Grace LPDDR5X act as a fast tier-2 KV pool. To verify it's actually being used at runtime:

```bash
# while serve is running with host_cache_size set, watch:
nvidia-smi dmon -s u    # check GPU memory utilization stays bounded
sudo numastat -m        # confirm Grace node memory grows as KV evictions happen
```

If `host_cache_size` is set but `numastat` shows no Grace memory growth under load, the host-offload path isn't firing — check `kv_cache_config.onboard_blocks` and that the version supports SBSA host-offload (v0.21+ added the path, v1.2.1 is current).

Sources:

### Reading the table

- **Batch 1, FP8 + EAGLE3:** This is the headline configuration for chat-style latency. ~90-150 tok/s single-user puts Llama-3.3-70B on GH200 in the same league as small-model cloud APIs.
- **Batch 64, FP8:** This is your steady-state-throughput sweet spot. Expect 2,000-3,300 tok/s aggregate, ~30-50 tok/s per stream.
- **Batch 256:** GH200-96 is genuinely tight here. KV pool calculus: 70B × 2 bytes/param FP8 ≈ 70 GB weights, ~25 GB free for KV, ~250 KB/token KV at GQA=8, ÷ 256 batch ≈ ~400 tokens of context per request. Realistic max useful batch on GH200-96 with non-trivial context is **~96-128**, not 256. Use 2× GH200 NVL2 if you actually need batch 256 at meaningful context lengths.
- **INT4 AWQ wins** at batch 1 (memory bandwidth dominates), **loses** at batch ≥ 16 (compute dominates and FP8 tensor cores rule). Confirmed by NVIDIA's own quant blog: AWQ "high perf" for batch ≤ 4, "low perf" for batch ≥ 16.

## 8. Final recommendation — what to actually run

If your team's goal is **"highest tok/s + least friction running Llama-3.3-70B 4-bit-class on TRT-LLM 1.2.1 GH200,"** the honest answer is:

1. **Drop the strict 4-bit constraint.** The Round 3 bf16-default policy was scoped to DiT/Wan, where FP8 is genuinely broken. For Llama-3.3-70B on Hopper SM90 *LLM* inference, FP8 via `nvidia/Llama-3.3-70B-Instruct-FP8` is the operational path NVIDIA actively maintains. The known FP8 bugs (Qwen2.5 KV gibberish, Llama PP+FP8 hang) **don't affect** single-card GH200 Llama-3.3-70B with FP8-weights-only (FP8-KV is also OK on Llama, just keep an eye on accuracy regression in your test suite).
2. **Container:** `nvcr.io/nvidia/tensorrt-llm/release:1.2.1` (arm64 manifest works).
3. **Model:** `nvidia/Llama-3.3-70B-Instruct-FP8`.
4. **Backend:** PyTorch (default in 1.2.1). No engine compile.
5. **Spec-decode:** EAGLE3 via `nvidia/Llama-3.3-70B-Instruct-Eagle3` for batch ≤ 32 workloads. Disable for batch ≥ 128.
6. **KV cache:** `enable_block_reuse=true` (when not using EAGLE3) + `host_cache_size=200GB` to exploit GH200's Grace LPDDR5X.
7. **`max_batch_size=64`, `max_num_tokens=8192`** as the GH200-96 baseline.
8. **If you must do 4-bit:** ModelOpt `int4_awq` → `trtllm-build` (§3). Budget ~1-2 hours wall-clock for first token. Expect to lose throughput vs FP8 at batch ≥ 16. Validate accuracy on 200+ samples vs bf16.

**Friction-weighted final ordering for "Llama-3.3-70B on GH200 TRT-LLM 1.2.1":**

| Rank | Path | Friction | Tok/s headline (batch 32, est) |
|---|---|---|---|
| 1 | **FP8 PyTorch-backend + EAGLE3** | **2** | 1,500-2,500 aggregate / 90-150 single-user |
| 2 | FP8 PyTorch-backend (no spec-decode) | 2 | 2,000-3,300 aggregate |
| 3 | BF16 PyTorch-backend | 2 | 1,200-2,200 aggregate |
| 4 | INT4 AWQ engine-build | 4 | 1,500-2,400 aggregate |
| 5 | "4-bit on PyTorch backend" | 5 | not a real option on 1.2.1 |
